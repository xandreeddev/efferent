## In-app `:login` (no wizard, no init, no env keys)
- The TUI always boots; credentials are added in-session and live **only** in `~/.efferent/auth.json` (the `AuthStore` port). `loginFlow.ts` is a **pure state machine** composing `selectBox.ts` (auth-method + provider steps, status-tagged) and `promptBox.ts` (masked API-key / pasted-redirect input); the driver (`modes/tui.ts`) runs the effects on each `LoginAdvance` outcome.
- Because the router resolves the key **per request** from the `AuthStore` (no layer-build capture), a login takes effect immediately — no restart. The first login also pins that provider's default model.
- OAuth subscription (Anthropic): `login/oauthServer.ts` opens the browser + runs a loopback callback server (races a manual paste of the redirect URL); `adapters/src/auth/oauth/anthropic.ts` holds the PKCE/exchange/refresh protocol; the router sends a Bearer token + the Claude-Code system block for `oauth` credentials.

## Activity pane (the default side view)
- The side pane's `stack` view is titled **activity** — an at-a-glance dashboard (`sidePane.ts`): a session-stats header (context-window gauge + cumulative `tok out` + turns + elapsed), the live execution tree (each `turn` node shows its per-LLM-call output `N tok` + duration, via `treeTurnDetail` fed from `assistant_message.usage`), and three **foldable sections** — `files` (a running diffstat accumulated from edit/write tool results), `skills` (seeded from the skills discovered at startup), `instructions` (the discovered `AGENT.md` files). Sections collapse to a one-line summary that still shows the key info (e.g. `▸ files (3) +24/-5`).
- **Navigable** with its own block cursor (`stackCursor`): `j/k`·`gg/G` move, **`Tab`/`Enter`/`h`/`l`** fold the row under the cursor — a section header **or** a tree **turn/sub-agent** node (`foldId` is the section name or `node:<id>`; `stackCollapsed` holds both). A collapsed turn shows `▸ turn N · M tools · <tok> <dur>` and hides its tool rows. **Auto-collapse**: when a new user message starts a run, the driver folds every prior turn/sub-agent container node (`node:<id>`, walked from `tree.roots`) into `stackCollapsed`, so the new run begins with its own steps expanded and the previous run tucked away (manual folds within a run persist; `stackCollapsed` resets to the section defaults when the tree resets on build/resume). `stackRows`/`stackModel` are the single source of truth for render + nav (row count + `foldId`s are width-independent); `render.ts` draws the block cursor + windows to follow it when focused, else bottom-anchored.

## Handoff & context viewer
- **`:handoff`** runs `createHandoff` (core) — summarizes the loaded view, writes a checkpoint, pushes a `checkpoint` scrollback block. **`:context`** toggles the side pane between the live `stack` view and the `context` view; **`:browse`** lists workspace conversations, **`:resume <#|id>`** switches to one (full replay via the shared `replayHistory` helper; execution then loads only the active window).
- `tui/contextView.ts` owns the viewer: `buildContextView(messages, checkpoints)` (pure, unit-tested) partitions into *archived* (folded, dim, not loaded — each carries its checkpoint `summary`) and *loaded* (bright, headed by the `✦` summary) segments; `buildContextRows(segments, collapsed, selected, selectedHandoffs)` groups each segment's messages into **turns** (a user message + its assistant/tool replies — the same "commit" unit as the conversation pane) and flattens segment → turn → message into navigable `ContextRow`s (turn rows carry a global `turnIndex` + the turn's first `messageIndex`; archived segment rows carry a `handoffIndex` and show a `✦` summary preview; a running `messageIndex` === conversation position); `renderContextView(rows, …)` draws the tree. Turns open folded (a clean, selectable list of subjects). The fold-point semantics live in `@efferent/core` (handoff/checkpoints) — this is display only.
- **The context viewer is interactive** (`sidePane.ts`): it has its own block cursor (`contextCursor`), fold set (`contextCollapsed`), and two **selection sets** — `contextSelected` (a set of `turnIndex`) and `contextHandoffSelected` (a set of `handoffIndex`). When the side pane is focused in `context` view: `j/k`·`gg/G` move the cursor, `Tab`/`h`/`l` fold a turn or handoff segment, **`Enter` jumps** the conversation cursor to the turn/message (`scrollback.cursorToMessageIndex`, fed by `msgIndex`-tagged blocks from `replayHistory`), **`Space` selects/deselects the turn *or* handoff** under the cursor (`◉`/`○` marker; the header shows an `N selected` count), and **`b` (or `:build`) builds a new session** from the selection. A handoff and its own inner turns are kept **mutually exclusive** at selection time (`sideToggleSelect` uses `archivedTurnRanges`/`handoffOwningTurn`): selecting the handoff drops its turns, selecting an inner turn drops the handoff. Pure nav helpers (`sideCursorMove`/`sideToggleNode`/`sideToggleSelect`/…) re-derive rows each call; `z` zooms for a fuller read.
- **Build-a-session** (`messagesForSelectedTurns` + the `doBuildSession` driver): collects the picked turns' messages in order (turn granularity keeps tool-call/result pairs intact) and, for each selected handoff, one `handoffToMessage(summary)` standing in for the whole archived segment (the compact summary instead of the folded originals) — landing as that segment's prefix, matching `runAgent`'s `[handoff prefix, …messages]` shape. Then `ConversationStore.create`s a fresh conversation, appends them, and switches the TUI to it (clears + `replayHistory` + focus the input) — mirroring the `:resume` switch. The original conversation is untouched.

## Hard rules
- No domain logic. If something looks like a decision about *what* the agent does (vs *how* it's invoked from a terminal), it belongs in `@efferent/core`.
- Each mode is a single `Effect.Effect<void, never, FileSystem | Http | Shell | LanguageModel | ConversationStore | SettingsStore | WebSearch>` (TUI adds `ModelRegistry` + `LlmInfo`) that subscribes to the agent's event queue and renders its way. (`WebSearch` is required because the coding toolkit's `web_search` handler resolves it.)
- `main.ts` is the *only* place adapter selection happens. To swap providers, swap the Layer imported here.
- Mode resolution defaults: argv prompt or piped stdin → print; TTY → tui; else print. `--mode <x>` overrides.
- `--help` and `--version` are provided by `@effect/cli` — don't shadow them.

## TUI invariants
- Regions, top → bottom: **two separate bordered boxes** (`┌─ conversation ─┐ ┌─ context ─┐` — titles in the top border) with **one empty column between them**; each pane has a **distinct accent colour** when focused — conversation **bright cyan**, side **bright magenta**, input **bright green** (`PANE_ACCENT` in `render.ts`; unfocused = dim gray), which is the primary focus cue. Then the **bordered keybind box** (`legend.ts`, a box whose border + title take the *focused* pane's accent; the title carries `<pane> · <MODE>`; the body is **two labelled rows** — a dim `nav` row of global movement keys (pane switching / `:` / `/` / zoom, identical in every pane) over a dynamic row of the focused pane's own keys, the "fixed instructions". A `:`/`/` entry takes the top row as `cmd`/`find` and blanks the bottom one), the `:` command palette (when open), the **input box** (its own bordered box, `┌─ input ─┐`, 1–8 content rows; title morphs `input`/`command`/`search`, border brightens green when focused), the **status bar** (one unboxed row: `model · token gauge · storage · cwd` — `storage` is the active store `sqlite`/`pg`; **no** mode/pane, since the mode now lives in the keybind box title), and a dim **footer** (`logs: …` path + key hints). Side box hidden < 60 cols (single box). The old top hint bar + pinned title row are gone (`header.ts` is dead).
- The input is a vim-style **command line**: its prompt morphs with `entry` — `❯` message · `:` command · `/` search (`input.ts:PROMPTS`). `:`/`/` open from NORMAL, or from INSERT on an empty buffer (so they stay literal mid-message); the body is typed in the input (no prefix char). Search shows a right-aligned `i/total` count on the input row; there is **no separate search bar**.
- **Submit is a NORMAL-mode action.** In INSERT, Enter (and Ctrl-J) insert a **newline** — composing never submits. You send a message by leaving INSERT (`Esc`) and pressing **Enter in NORMAL** (`viMode.ts` handles NORMAL Enter→submit; `input.ts`'s Enter case is newline-only and is the INSERT path). Empty input doesn't submit. (`:`/`/` Enter still run/jump via the command-line `entrySubmit`.)
- **Real nvim block cursor** (the single "where am I" anchor — NOT a caret glyph): `scrollback.ts` carries a 2D `(cursorLine, cursorCol)` + a remembered `desiredCol`. It's drawn as the **hardware terminal cursor** (`render.ts` does `moveTo` + `showCursor`): a steady **block** (`cursorBlock`) on the focused pane in NORMAL/VISUAL — including the input line in NORMAL — and a **bar** (`cursorBar`) only in INSERT (the input's only non-NORMAL mode) (DECSCUSR shapes, reset on exit). A dim full-row tint (`terminal.ts:bgCursorLine`) marks the cursor line. Content sits after a 2-col left margin inside the box, so the cursor's screen cell is `boxLeftBorder + gutter + cursorCol`. `cursorLine` is the source of truth — every motion moves it and the viewport follows with a scrolloff margin (`followCursor`, which also `clampCol`s); `scrollOffset` is derived. Conversation NORMAL motions (all move the block cursor): `h/l` char, `w/b/e`·`W/B/E` word, `0/^/$` line, `j/k` line (keeping `desiredCol`), `Ctrl-D/U` half-page, `PgUp/PgDn` page, `gg/G` ends, `{/}` **turn** hops. The side context-tree pane has its own block cursor (`sidePane.ts`). `:` opens the command line + palette.
- **Follow-tail**: a cursor on the last line rides new streaming output to the bottom; a cursor parked up in history is *not* yanked down by appends (`render()` snapshots `wasAtTail` before re-flatten, and bumps the offset by the growth otherwise). `submit()` re-engages the tail (`cursorToBottom`). `initCursor()` (called on focus-in) parks a never-placed cursor at the newest line; a placed cursor persists across focus changes.
- **Search returns to the conversation**: `/` is typed in the input line, but on Enter focus lands on the **conversation pane** (NORMAL) with the cursor on the match — `n/N` then move the cursor between matches. Esc clears the highlight and returns to whichever pane opened the search (`preSearchFocus`). The conversation caret stays visible (dim) while you type the query.
- **Zoom** (`z` in NORMAL on a read-only pane, `uiMode.ts` focus unaffected): maximizes the focused pane to fill the whole middle region (other pane + divider hidden) for distraction-free reading; `z` or Esc exits, and swapping focus / opening the command line auto-exits (you can never be zoomed on a hidden or input pane). Header / input / status / title rows stay; only the middle collapses. (`Ctrl-w` is intentionally unused — it'd need a chord state machine; single-key `z` is collision-free.)
- **Modal + multi-pane** (vim-flavoured, always on; no `editorMode` toggle). Three focusable panes — conversation / side / input — swapped with **Ctrl-h/j/k/l or Ctrl-arrows** in *any* mode (`uiMode.ts:moveFocus`); a swap only fires when a pane exists that way, so Ctrl-J/Ctrl-H still act as newline/backspace in the input and a non-moving Ctrl-arrow falls through to that pane's in-pane motion (nothing is below/left of the input). Ctrl on arrows is decoded from the xterm modifier in `keys.ts:parseCsi` (`CSI 1;5A`). Entering the **input** pane → INSERT (type immediately); entering a read-only pane → NORMAL. Per-pane modes: **INSERT** only on the input pane; **NORMAL + VISUAL** on the read-only panes. Start: input focused, INSERT.
- **Arrows are peers of hjkl** everywhere keys move: conversation ↑/↓ scroll · ←/→ char cursor · Home/End line ends; side pane ↑/↓ cursor · ←/→ fold a node (mirroring h/l). Plain arrows never switch panes — only Ctrl-arrows do.
- **All key routing is a pure function** — `navKeys.ts:decideKey(ctx, key) → NavIntent` (no Effects, no mutation); the driver executes the intent. Unit-tested in `navKeys.test.ts` (`bun test`). Active-pane signal: the focused box's border + title brighten to that **pane's accent** (conversation cyan / side magenta / input green), and the keybind box mirrors the same accent + names the pane in its title (`<pane> · <MODE>`).
- **Neogit-style foldable sections** (`scrollback.ts:flatten`): the conversation is a tree of **turns** ("commits") — a `user` block + the assistant/tool/reasoning blocks until the next user/checkpoint. Each turn is a foldable handle (`▾`/`▸` + green `┃` subject); folded → one summary line (`▸ ┃ <prompt> · N blocks`). Inside a turn, a run of ≥2 consecutive tool calls folds into one group (`▾ N tool calls` / `▸ ⊞ N tool calls`). `Tab`/`Enter` fold the section under the cursor, `Z` folds/unfolds all; section ids are stable (a monotonic `seq` stamped per block, preserved across `updateTool`) so a folded group stays folded as streaming pills append. `flatten` records, at post-fold positions, `msgStarts` (turn/assistant starts for `{`/`}`), `itemAtLine` (line → foldable section), and `msgIndexToLine` (jump targets). `checkpoint` blocks are top-level dividers (not folded).
- **Block styling** (`scrollback.ts:renderBlock`): an **assistant** turn gets a cyan `●`; a **reasoning** block (the model's externalised thinking, surfaced from the response's reasoning parts via `assistant_message.reasoning`) renders dim+italic with a quiet `✻`, capped to ~4 lines until Ctrl-R. The model is nudged (coder prompt) to write a one-line intent before tool calls, and the loop emits `onAssistantMessage` *before* that step's tool events so narration/reasoning renders above its tool pills. `edit_file` diffs use coloured +/- text with **no filled background**. Badges are black-on-bright (`terminal.ts:bgBright*`) for contrast.
- Keys (parsed in `keys.ts`): Backspace is `0x7f` and Enter is `0x0d` only, so `0x08`/`0x0a` surface as **Ctrl-H/Ctrl-J** (needed for pane focus); insert-mode Ctrl-H still acts as Backspace via `input.ts`.
- VISUAL (conversation pane): **`v` charwise** / **`V` linewise**, anchored at the current `(line,col)` cursor; the same block-cursor motions extend it. `y` yanks the selection — charwise = the exact spanned cells, linewise = whole lines — to the clipboard via **OSC 52** (`terminal.ts:osc52`); Esc/`v` cancel.
- **No mouse tracking** is enabled — the TUI stays out of mouse-reporting mode so terminal-native click-drag selection keeps working; copying is also available keyboard-side via VISUAL + `y`.
- `Scrollback` memoizes per-block wrapped lines (keyed by block identity + cols + expanded) so a keystroke/spinner tick re-flattens cached arrays instead of re-wrapping. Above/below "↑ N above" / "↓ N below" indicators are independent (top and bottom rows).
- Side pane shows the live agent stack (parent + any in-flight sub-agent + its current tool), skills loaded this session via `read_skill`, and AGENT.md files discovered at startup. Hidden under 60 cols. The divider leans bright toward the focused side.
- Sub-agent inner tool calls do NOT push pills to scrollback — they update the side pane's top frame's `currentTool`. The parent's `delegate_to_<name>` pill stays in the scrollback.
- Renders are full-frame composed then line-diffed against the previous frame to avoid flicker.
- Raw mode + alt buffer + bracketed-paste; restored on exit (Ctrl-C, `:exit`, signal).
- Bash safety: non-interactive modes gate on `--allow-bash` via the `allowBash` flag passed to `codingToolkitLayer`; a denied call is returned to the model as a tool failure. The TUI currently allows bash unconditionally — a `promptForBash` modal is defined in `tui.ts` but **not yet wired** into `onBeforeToolCall` (see `docs/roadmap.md`).

## Hardcoded knobs (move to a settings layer later)
- Bash timeout default: 60s.
- TUI palette: 6 visible rows; `:` commands hardcoded in `slashPalette.ts`. Hint bar: 1 row. Wheel tick / page step live in `scrollback.ts`.
- `maxSteps` for the agent loop: 20.
