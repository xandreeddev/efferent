# efferent — agent runtime on Effect.ts + Bun

An **agent runtime on Effect.ts** — a pure-domain SDK (agent loop, context tree, compaction, multi-provider router, approval — all composable Effects/Layers) plus **one CLI that runs agents** (an interactive TUI, headless print / json / rpc, or a persistent per-workspace daemon that thin clients attach to). The bundled coding agent is just one app the runtime runs (reached via `efferent code`). Built in public as `@xandreeddev`. See `../CLAUDE.md` (parent tree) for the broader project rules — alias identity, OPSEC, weekly cadence, locked stack. See `docs/roadmap.md` for what's deferred and `docs/models.md` for the exhaustive model-usage map (every LLM call site, its selection, where the spend lands).

## Architecture (ports & adapters)

```
packages/
├── core/         pure domain: entities, ports, use cases, prompts — depends on `effect` + `@effect/ai`
├── adapters/     Layer impls of ports — depends on @xandreed/sdk-core + external SDKs (@effect/ai-google, @effect/ai-openai, Postgres)
└── cli/          the efferent CLI driver: runs agents — TUI · print · json · rpc · daemon
```

**Dependency direction is strictly inward.** `cli` → `adapters` → `core`. `core` imports nothing from siblings. `adapters` imports `@xandreed/sdk-core` + the external SDK it wraps. Drivers compose the layers at the very edge and hand off to `BunRuntime.runMain`.

## Conventions

- **Ports** are `Context.Tag` services in `@xandreed/sdk-core/ports/`. Each ships its tagged errors next to it.
- **Adapters** provide one `Layer.effect` per port. External promises go through `Effect.tryPromise`, mapped into the port's tagged error. Config via `Config.string` — never hardcode.
- **Use cases** live in `@xandreed/sdk-core/usecases/` returning `Effect.Effect<A, E, …>`. No IO; the only SDK allowed in `core` is `@effect/ai` (provider-agnostic — `LanguageModel`, `Tool`, `Toolkit`, `Prompt`). Provider packages (`@effect/ai-google`, `@effect/ai-openai`) live in `adapters`.
- **Agent configs** (`@xandreed/sdk-core/usecases/{agentConfig,coderAgentConfig}.ts`) bundle a system prompt + an `@effect/ai` `Toolkit` into an `AgentConfig<Tools>`. `runAgent` is parameterized by config; the CLI picks `coderAgentConfig(cwd)`. The toolkit's handler `Layer` (`codingToolkitLayer(cwd, skills, { allowBash })`) is provided at the driver's composition root — it carries the runtime deps (`cwd`, `FileSystem`, `Shell`).
- **Schema** lives in `effect` itself: `import { Schema } from "effect"`.
- Bun runs `.ts` directly. No build step, no emit. `tsc --noEmit` is purely a typecheck gate.
- **No `try/catch`, `throw`, or `.catch()` in `@xandreed/sdk-core/src/`.** Error handling uses Effect's typed errors (`Effect.fail`, `Effect.die`, `Effect.catchAll`, `Effect.catchTag`). Enforced by `scripts/banTryCatch.ts` — a zero-dep TypeScript-AST scan (so `Effect.try`/`Effect.tryPromise`/strings/comments never false-positive), folded into `bun run typecheck`.
  - **After the task is complete, run `bun run typecheck` to validate the ban.** It runs tsc + the AST scan; a banned construct fails the command and the change is rejected.
  - Enforced at three layers: a local `pre-commit` hook (installed by `bun install` via the `prepare` script → `scripts/installHooks.ts`; `--no-verify` bypass discouraged), the `ci` workflow on every push/PR (`.github/workflows/ci.yml`), and the release gate. Branch protection on `main` requires `ci`, so a violation can't be merged or pushed.
- File naming: camelCase for files that export functions; PascalCase for files that export types / `Context.Tag` classes.

Credentials & setup. **Credentials come *only* from `~/.efferent/auth.json`, written by the in-app `:login` flow — there is no env-var key reading and no `init` command.** `efferent` always boots straight into the TUI; with no credential it shows `No models available. Run :login …` and sending a message short-circuits to the same hint. **`:login`** is an in-app flow (`packages/cli/src/cli/presentation/loginFlow.ts` + `promptBox.ts`, composited like the `:model` picker): pick *Use a subscription* (OAuth) or *Use an API key* → pick a provider (status-tagged ✓api key / ✓subscription / •unconfigured) → paste a masked key **or** run the OAuth flow → persisted to `auth.json` → **usable that turn, no restart**. **`:logout <provider>`** removes a credential. The first successful login also pins that provider's default model. Non-interactive modes (print/json/rpc) can't run `:login`, so they require a credential already in `auth.json` (else they exit with a hint).
- **`AuthStore`** port (`@xandreed/sdk-core/ports/AuthStore.ts`) + `LocalAuthStoreLive` (`adapters/src/auth/local.ts`): per-provider creds (`{ type: "api_key" } | { type: "oauth", … } | { type: "local", baseUrl? }`; legacy flat-string entries read as api keys), atomic `0600` writes. `resolveKey` is read lazily per request by the router/registry, refreshing a near-expiry OAuth token first. Reads no env. **Config/auth scoping** (shared with `SettingsStore`, resolved by `adapters/src/configRoots.ts`): **`EFFERENT_HOME` set ⇒ that one dir is the entire world** — `$EFFERENT_HOME/.efferent/{auth.json,config.json}`, no `~/.efferent`, no cwd, no merge (the hermetic sandbox / `start:fresh`). **Unset ⇒ two tiers** — global `~/.efferent` + local `<cwd>/.efferent`, merged **local-over-global**; onboarding picks the scope (step 1), `:login` defaults global, `:set`/`/model`/`:theme` default local, and a local write seeds a `.efferent/.gitignore` (`auth.json`, `config.json`, `*.db*` — `skills/` stays committable). The router builds the provider client per turn from `resolveKey`, so a login mid-session takes effect immediately. Evals/CI use `EnvAuthStoreLive` (`adapters/src/auth/env.ts`) — the *only* place provider key env vars are read.
- `EFFERENT_DB_URL` — **optional.** Selects the conversation store: `postgres://…` → Postgres; any other value → SQLite at that path (optionally `sqlite:`-prefixed); unset → zero-config **SQLite** at `~/.efferent/efferent.db` (the default; no Docker). May also live as `dbUrl` in `~/.efferent/config.json`; the env var **overrides** the config value (seeded into the env at boot by `seedDbUrlFromConfig`, since the store is chosen at layer-build before settings load). Parsed by `parseDbTarget` and selected in `adapters/src/database/migrator.ts` (`ConversationStoreLive`); migrations load via `Migrator.fromRecord` (bundle-safe).
- Optional `EFFERENT_MODEL` — seeds the active model when no `config.json` pins one. Accepts `"<provider>:<modelId>"` (e.g. `openai:gpt-4o`) or a bare id (provider inferred; defaults to Google). Default `google:gemini-3.5-flash`. An explicit `/model` switch is persisted to `.efferent/config.json` and wins over `EFFERENT_MODEL`.

## Install / distribution

Published as the unscoped npm package **`efferent`** (primary) and ALSO as the scoped alias **`@xandreed/cli`** (same bundle, kept in sync); the previously published scoped package is deprecated. Bins are just **`efferent`** / **`eff`** (the old `e` and `code` bins are gone). It's a Bun bundle (`bun run build` → `packages/cli/dist/efferent.js`, core + adapters + `@effect/*` + `@opentui/solid` + `solid-js` inlined, `#!/usr/bin/env bun`). Requires **Bun** at runtime and has **two runtime dependencies: `@opentui/core`** (+ its platform-native `@opentui/core-<platform>` subpackage) — the TUI's native Zig renderer is `dlopen`'d via FFI and can't be inlined, so it stays external; it *also* ships the tree-sitter grammar WASM (`assets/{javascript,typescript,markdown,zig}`) + the `parser.worker.js`, both resolved from its own install dir — **and `web-tree-sitter`** (the highlighter that worker imports; a peer dep of `@opentui/core`, so we declare it directly or `npm i -g` won't install it). Everything else is inlined; the workspace + the other `@effect/*` are devDependencies. `npm i -g efferent` then just run `efferent` — it boots into the TUI and you add a provider in-session with **`:login`** (subscription/OAuth or API key); no wizard, no `init`, no env vars. Works in any project with no Docker (SQLite history + `auth.json` under `~/.efferent/`). **Live on npm since 2026-06-12** (current CLI `0.5.0`), publisher `xandreed` (the alias npm account). **The SDK libraries are published too**, independently versioned: `@xandreed/sdk-core` (pure, runtime-agnostic), `@xandreed/sdk-adapters`, and `@xandreed/evals` (the reusable framework only — its CLI-coupled suites/`runCoder` stay in-repo). The libs ship compiled **JS + `.d.ts`** (`bun run build:libs` → `tsc -b` project references → `packages/*/dist`); dev still runs raw `.ts` (their `exports` point at `./src`, resolved via the workspace + tsconfig `paths`, no build). **Releases are Changesets-driven — no manual version bump, no `gh release create`**: add a changeset (`bun run changeset`) in your PR → on merge to `main`, `.github/workflows/release.yml` (`changesets/action`) maintains a **"Version Packages" PR** (bump + CHANGELOGs); **merging that PR publishes everything in ONE run** — the action runs `bun run release` (= `scripts/publish.ts`) as its `publish` command (there is NO manual per-package step; the `publish-*.yml` workflows were removed). `scripts/publish.ts`: a cheap `npm view` pre-pass finds the versions not yet on npm, builds only those, then `npm publish --provenance`es each (npm **trusted publishing** / OIDC, no token — each published name's Trusted Publisher on npmjs.com must point at **`release.yml`**), rewriting each lib's manifest at publish time (`exports`/`main`/`types`→`dist`, `workspace:*`→real `^` ranges) then restoring the dev manifest, and still dual-publishes the CLI (`efferent` + `@xandreed/cli`). A brand-new scoped name needs ONE manual bootstrap publish first (OIDC can't create it) — see `CONTRIBUTING.md`. Private packages (`social`/`website`/`examples`/root) are auto-excluded. The build stubs the provider tokenizer packages out of the bundle (`scripts/build.ts` — they baked the build machine's absolute path into the artifact and would crash a published install at module-init; efferent never tokenizes locally). Cutting a release is outward-facing — only on explicit sign-off.

## Coding tools (CLI)

An `@effect/ai` `Toolkit` in `@xandreed/sdk-core/usecases/codingToolkit.ts`, backed by the `FileSystem` and `Shell` ports. Each tool is a `Tool.make` def with an **object** `success` Struct + a shared `failure` Struct and `failureMode: "return"` (so a tool failure is returned to the model as data, not thrown). Handlers live in `codingToolkitLayer(cwd, skills, { allowBash })`, which resolves `FileSystem`/`Shell` from context at layer-build time. (Gemini rules learned the hard way: every tool needs ≥1 parameter, and `success` must be an object — see the spike notes in the plan.)

| tool         | parameters                                       | implementation                              |
|--------------|--------------------------------------------------|---------------------------------------------|
| `read_file`  | `{ path; offset?; limit? }`                      | `FileSystem.read`                            |
| `write_file` | `{ path; content }`                              | `FileSystem.write`                           |
| `edit_file`  | `{ path; edits: [{ oldText; newText }] }` *or* flat `{ path; oldText; newText }` | `normalizeEdits` → read → string replace → write, returns diff |
| `Bash`       | `{ command; timeout?; run_in_background? }`       | `Shell.exec` (cwd bound at tool-build time); default timeout **5 min** (`DEFAULT_BASH_TIMEOUT_MS`). `run_in_background:true` → `Shell.spawnBackground`, returns `{ processId; running }` at once |
| `bash_output`| `{ processId; sinceCursor? }`                    | `Shell.readBackground` — incremental stdout/stderr + exitCode/running of a background proc |
| `kill_bash`  | `{ processId }`                                  | `Shell.killBackground` — group-kills a background proc |
| `session_start` / `session_send` / `session_read` / `session_kill` / `session_list` | tmux session id + keys/lines | `TerminalSession` (tmux): start a detached interactive pane (TUI/REPL), send keys, capture screen, kill; namespaced so a human can `tmux attach`. Feature-detected; no tmux ⇒ graceful failure |
| `grep`       | `{ pattern; dir?; flags?; context? }`            | shells out to `grep -rnE` for now            |
| `glob`       | `{ pattern; dir? }`                              | `FileSystem.glob` via `Bun.Glob`             |
| `ls`         | `{ path?; recursive? }`                          | `FileSystem.list`                            |
| `web_fetch`  | `{ url; maxBytes? }`                             | `Http.get`; HTML reduced to text             |
| `search_web` | `{ query }`                                      | `WebSearch.search` → `{ answer; sources }`   |
| `run_agent`  | `{ name; folder; task; seedFromNode?; seedMode?; agent?; instructions?; tools?; role? }` | spawns a folder-scoped sub-agent **in the background** and returns `{ nodeId; name; status: "running" }` immediately — non-blocking (see Sub-agents & context tree); `name` (2-5 words) is its display title; `role` (`general`\|`code`) picks the model TIER, never a specific model |
| `wait_for_agents` | `{ nodeIds?; timeoutSeconds? }`             | the non-blocking **gather**: parks the caller (interruptibly) until a watched agent finishes / a message lands in the caller's inbox / the timeout, then returns each agent's status (+ finished summaries/files), the caller's drained inbox, and the blackboard tail. `AgentBus.awaitChange` |
| `update_plan` | `{ steps: [{ step; status }] }`                  | the agent's working plan (statuses pending/active/done); each call replaces it whole — pure echo handler, the UI reads the args |

All paths resolved relative to the `cwd` bound in `codingToolkitLayer(cwd)`. **`edit_file` accepts two shapes:** the canonical `edits: [{ oldText, newText }]` array, or a flat single-edit `{ oldText, newText }` at the top level — the shape models trained on Claude Code's `Edit` tool emit when there's only one edit. Both are normalised by `normalizeEdits` *inside* the handler. This matters because `@effect/ai` decodes tool-call parameters **before** the handler runs (`Toolkit.handle`), so a shape mismatch is an `AiError.MalformedOutput` that `failureMode: "return"` can't catch — it aborts the whole turn. Keeping the parameter schema permissive enough to decode both shapes, then validating in the handler, turns "wrong shape" into a graceful, model-visible `EditFailed` instead of a dead turn. **Names matter:** Anthropic reserves the lowercase tool names `bash` / `web_search` / `computer` for its provider-defined tools (case-sensitive lookup); when we register a handler-backed tool by one of those names, the Anthropic SDK reroutes it to the built-in provider tool — which isn't in our toolkit, so the turn fails. So our shell is **`Bash`** (capital B) and our search is **`search_web`** (reversed). `search_web` is provider-native (see below) — it has no parameter-less constraint issue because `{ query }` satisfies the ≥1-parameter rule.

## Web search

Provider-native, no extra key. The `web_search` tool is backed by the `WebSearch` port (`@xandreed/sdk-core/ports/WebSearch.ts`) and `WebSearchLive` (`adapters/src/llm/webSearch.ts`). Each call is a **dedicated, grounding-only** `generateText` against a provider's *server-side* search tool — Gemini `GoogleTool.GoogleSearch` or OpenAI `OpenAiTool.WebSearch` (`Tool.ProviderDefined`, handler-free). It returns `{ answer, sources }` (sources from the response's `UrlSourcePart`s). The model finds with `web_search`, then reads a chosen source with `web_fetch`.

**Deliberately a separate thing** (not merged into the agent's main toolkit): the search call carries *only* the search tool, never the agent's function tools — so it needs no extra credential beyond a logged-in provider, and sidesteps providers (notably Gemini) that won't combine grounding with function calling in one request. It's also decoupled from the chat `/model`: configured via `EFFERENT_SEARCH_MODEL` (`<provider>:<modelId>`), else defaults to whichever of Google/OpenAI is logged in (Google preferred). `WebSearchLive` resolves its key per call from the `AuthStore` and builds its own client over a `FetchHttpClient`. Gemini's grounding source URLs are `vertexaisearch…` redirects — `web_fetch` follows them.

## Agent loop

One use case — **`runAgent(config, conversationId, prompt, hooks?)`** in `@xandreed/sdk-core/usecases/runAgent.ts` — drives the whole interaction. The loop lives in `@xandreed/sdk-core/usecases/agentLoop.ts`. **`@effect/ai` resolves a single model step's tool calls (its handlers are our Effects) but does NOT iterate across turns — so iteration is ours**: each turn maps the message buffer to a `Prompt`, calls `LanguageModel.generateText({ prompt, toolkit })`, appends the response parts as the new tail, and re-invokes until `finishReason !== "tool-calls"` or `maxSteps`.

**The root has TWO shapes, keyed on `isOrchestrateMode(agents)` (a fleet lead in the roster) — and the prompt and toolkit branch on the SAME condition so they can never contradict each other.** When a `coordinator`/`research-coordinator` is loaded (the normal CLI, via `withBuiltinAgents`) the root is a **pure orchestrator**: it gets the **orchestration-only toolkit** — just `run_agent` (the hard-railed `RootDelegateTool`, locked to delegating to a lead), `wait_for_agents`, `send_message`, `update_plan`; **NO work tools** (no read/grep/glob/edit/Bash) — so it *cannot* do work itself and *must* route to a lead. Its system prompt is `orchestratorSystemPrompt` (`packages/cli/src/prompts/coder.ts`), a lean delegate-first prompt that lists only those four tools and never mentions reading the workspace or the blackboard. With NO fleet present (a bare roster), the root falls back to `directCoderSystemPrompt` + the full coding toolkit (the hands-on fast path). The toolkit gate (`buildScopeRuntime.ts`) and the prompt both call `isOrchestrateMode` — the single source of truth. (History: the original direct-coder root kept doing broad reads/edits itself instead of using the swarm, so #83 stripped its work tools as a mechanical guarantee; the prompt wasn't switched to match — it still advertised work tools and told the root to "read the blackboard FIRST" — and a weak model looped on the housekeeping tools (`list_scheduled_jobs`/`blackboard_read`) instead of delegating. The fix (2026-06-30): a dedicated orchestrate prompt that matches the stripped toolkit, the toolkit trimmed to the four essentials, and the loop breaker rewritten to catch interleaved/`update_plan`-churned no-progress spins.) The degenerate-loop circuit breaker (`agentLoop.ts`) is the backstop: a run of consecutive no-progress turns (a result-keyed signature already seen this run) nudges once, then force-stops.

```
efferent <prompt>
  ↓
runAgent(coderAgentConfig(cwd), conversationId, prompt, hooks)
  ├── ConversationStore.append(user) ; list → message history
  ├── runAgentLoop({ system, messages, toolkit, hooks, maxSteps })
  │     ├── turn 0..N: onTransformContext? → onTurnStart
  │     │              → Prompt.make([system, ...toPromptMessages(messages)])
  │     │              → LanguageModel.generateText({ prompt, toolkit })   (resolves this step's tools)
  │     │              → responseToAgentMessages(res.content) onto buffer  (carries thought_signature in providerOptions)
  │     │              → re-emit onBeforeToolCall / onAfterToolCall / onAssistantMessage(TokenUsage) from the response
  │     │              → continue iff finishReason === "tool-calls" && toolCalls > 0 ; onShouldStopAfterTurn?
  │     └── while turnIndex < maxSteps
  ├── ConversationStore.append(tail)
  └── return AgentResult { finalText, messages }
```

**Mandatory swarm gate — ONE tier, at the root** (`driveLoop` in `runAgent.ts`): when a run **used sub-agents**, the finished objective is submitted to the independent Opus gate (`Verifier.gate`) BEFORE `runAgent` returns — in `driveLoop`, the one path `runAgent`/`resumeAgent` (and thus every mode) funnel through, so a fan-out can't reach "done" without verification, regardless of whether a `coordinator` led it. **Fail-closed**: `needs_work` → re-run the loop with the gate's reasons fed back → re-gate, looping to `maxLoopAttempts`; an unavailable verifier (no `claude`) is surfaced **loudly** (a `gate` `AgentEvent` on the rail), never a silent pass. Gated by `autoLoop` (default on); a run with no sub-agents is untouched. **The old per-coordinator gate tier is GONE** (2026-07-02): it ran a multi-minute Opus subprocess inside the sub-agent stall watchdog's 180s window with no progress stamps, so finished leads were routinely killed as `[stalled]` and their work discarded (the top failure bucket in the July run forensics). Learning is also single-shot now — the turn-boundary distill (`runAutoDistill` in the drivers) is the ONE distillation per run; `gateOnce` no longer distills. Full detail in Self-improving loop → (E)/(F).

**Prompt mapping** (`@xandreed/sdk-core/usecases/promptMapping.ts`): bridges our persisted `AgentMessage` (our stable shape, unchanged) with `@effect/ai`'s `Prompt`/`Response`. The opaque provider blob is carried verbatim both ways (`providerOptions ↔ options`/`metadata`), which is how Gemini's `thought_signature` round-trips across our turns — `Prompt.fromResponseParts` drops it, so we map by hand.

**Hooks** (`@xandreed/sdk-core/AgentHooks`): the loop re-emits the legacy event vocabulary (`onTurnStart` / tool events / `onAssistantMessage`) from each resolved response, so the CLI's `makeEventHooks(queue)` (`packages/cli/src/events.ts`) and the TUI execution tree keep working unchanged.

**Graceful tool errors**: each tool's `failureMode: "return"` + `failure` Struct means a handler failure (e.g. `FileNotFound`, ambiguous edit) is returned to the model as a tool result instead of aborting the turn. **But `failureMode` only catches *handler* failures** — `@effect/ai` decodes a tool call's parameters (and resolves the tool name) *inside* `Toolkit.handle`, **before** the handler runs, so a wrong-shaped call or a hallucinated tool name fails with `AiError.MalformedOutput`, which `failureMode` never sees and which aborts the whole turn. `recoverMalformedToolCalls` (`agentLoop.ts`) closes that gap: it resolves the toolkit's handler once and wraps it so a `MalformedOutput` is converted into an ordinary tool *result* (`isFailure: true`, `error: "InvalidToolCall"`). The assistant tool-call ↔ tool-result pairing stays valid, the loop proceeds to its next turn, and the model reads the decode error from context and emits a corrected call — the **same** recovery path as any returned failure, no retry machinery. (`MalformedInput` — a result encode/validate failure, i.e. *our* bug — is deliberately let through, not masked.)

**Bash safety**: three layers. The static `allowBash` flag (denied → returned as a tool failure; non-interactive modes pass `--allow-bash`), then the **`Approval` port** (`@xandreed/sdk-core/ports/Approval.ts`) consulted per command, fronted by the **fast-tier auto-approval judge**. An unmatched command first goes to the judge (`judgeApproval` in `core/usecases/autoApproval.ts`, `UtilityLlm` on the **fast** role; `:set autoApprove off` disables, default ON): it classifies the input against the **permitted folders** — the workspace root (opening efferent in a cwd is the standing grant) plus every folder granted before — and either **allows silently** (ordinary development work whose paths stay inside; a dim `fast approved: …` info line in the rail, spend → `byRole.fast`) or **prompts**, the modal carrying its one-line reason. Not a security boundary (the scope sandbox owns that) — a prompt-fatigue killer: the judge can only remove prompts, never a human's view of anything it isn't sure about, and any error (no key, bad JSON, 429) degrades to the modal. The TUI modal (`cli/approval.ts`, `a/s/p/d`): **allow once / allow for session / always allow in this project / deny with a typed reason**. Permission is **path-based**: when the judge named the out-of-bounds folder, session/project answers grant *that folder* (session Ref / `Settings.approvedFolders`) and future commands inside it pass the judge; with no folder hint they grant the command *rule* as before (`bashRuleKey`: command+subcommand, exact-match when shell metacharacters are present — persisted to `Settings.approvedBashRules`). A denial (with its reason) returns to the model as an ordinary tool failure, so it adjusts in the same turn. Evals/CI provide the static allow-all impl behind the `--allow-bash` gate — CI never prompts and never judges. **Unattended (scheduled) runs are different** — they can't prompt a human, but blindly allowing everything is unsafe, so the cron path uses a **headless parking approval** (`makeHeadlessApproval`, `packages/cli/src/workspace/headlessApproval.ts`): the fast-tier judge still auto-approves ordinary in-scope work, but anything it can't clear emits a **`needs_human`** event (`parked: true`) and is **denied-with-reason** (returned to the model, never silently allowed). Interactive approval prompts emit the same event with `parked: false`, so a top-level "decisions need you" roster can surface both (see CLI shape → `DecisionsBar`). This replaces the old blind `ApprovalAllowAllLive` on the cron path.

## Sub-agents & context tree

Sub-agent spawning is **one generic tool over a persistent branching context tree**, and it is **asynchronous** — the fleet runs like a team of people, not a blocking call tree. **The whole orchestration substrate now lives in `@xandreed/sdk-core`** — `buildScopeRuntime.ts`, the unified `agentBus.ts`, `codingToolkit.ts`, `discoverScopeTree.ts`, `loadTools.ts`, `schedule.ts`, `staleness.ts`, and `parseFrontmatter.ts` are all `core/usecases/`, and the shared prompt pieces (`sections.ts` = `subAgentsSection`/`coordinationSection`/`renderAgentsSection`/`renderMemorySection`; `scopeAgent.ts` = `renderScopeSystemPrompt`) are `core/prompts/`. The loop primitives they call (`runAgentLoop`, `RunContextRef`, `tokenBudget`) are core too. The bus is the **Supervisor** — the orchestration substrate, owned by core (there's an exported `type Supervisor = AgentBus`). It is **threaded as a per-session value, NOT a `Context.Tag` port**, because it's per-session stateful and carries a per-runtime event sink (`onBusEvent`) — a single shared Layer would fight that.

- **`run_agent({ name, folder, task, seedFromNode?, seedMode?, agent?, instructions?, tools?, role? })`**: spawns a sub-agent scoped to `folder` (writes + bash confined via `ScopeBinding{rootDir, enforceWrite}` — the sandbox machinery in `codingToolkit.ts`, unchanged) and **forks it as a supervised background fiber** (`Effect.forkDaemon`, registered on the bus), returning `{ nodeId, name, status: "running" }` **immediately** — the caller never blocks on the subtree (that was the "parent hangs while the fleet works" bug). `failureMode: "return"` so a failed spawn is data, not a dead turn. The toolkit is **static** (`genericToolkit` = base tools + comms + `run_agent` + `wait_for_agents`) — the same toolkit at every depth; sub-agents can spawn sub-agents.
- **Gather, don't block** (`wait_for_agents`, `AgentBus.awaitChange`): a spawner collects results with `wait_for_agents({ nodeIds?, timeoutSeconds? })`, which parks **only the caller's fiber**, interruptibly, until a watched agent finishes, a message lands in the caller's inbox, or the timeout — then returns each agent's status (+ finished summaries/files), the caller's drained inbox, and the blackboard tail. No agent ever blocks another; loop it until `allDone`. The timeout default is **10s** (was 60s; clamped 1–300) — the bus wakes the waiter instantly on real progress (a finish / an inbox message), so the timeout is just the idle ceiling and a tighter one keeps a stuck waiter responsive. When a run finishes, the bus also drops a completion line in its **parent's mailbox** + the blackboard and keeps a terminal result record, so a parent that wasn't waiting still picks it up at its next turn boundary.
- **The bus is the live layer** (`agentBus.ts`, `AgentBus`): per-agent **mailboxes** (keyed by node id; the root conversation registers one under its `conversationId`, so the human can message the lead mid-turn), a shared **blackboard**, plus **supervision** — each running agent's fiber (so `:stop`/Esc/teardown interrupt it via `interruptAll`), a completion `Deferred` (so a parent awaits without polling), parent→child links (`childrenOf`, running + finished), and a `snapshot`. The durable record stays the context tree; this is the live layer the TUI + agents tap (`:fleet`, `:tree`, the header `◆ N agents` chip, a node preview).
- **The built-in team** (`teamAgents.ts` in `packages/cli/src/usecases/`, merged by `withBuiltinAgents`): when the root *does* decide to delegate (the opt-in triggers above), it hands the work to a **`coordinator`** (leads — `run_agent` + `wait_for_agents` + comms, no writes), which staffs the specialists the work needs — **`frontend` / `backend` / `qa` / `product` / `implementer`** (leaf coders + comms) — and validates each piece with the read-only **`architect`** (fresh context, SOUND/NEEDS WORK/BLOCKED) before delivering. The team is unchanged from before — only the *trigger* changed (delegation is opt-in, not mandatory). A workspace `.efferent/agents/<name>.md` overrides any built-in.
- **Persistence** (`ContextTreeStore` port, `@xandreed/sdk-core/ports/ContextTreeStore.ts`; SQLite migration `0002` / pg `0006` + the outcome migration sqlite `0010` / pg `0014`, tables `context_nodes` + `context_messages`): every spawn is an `AgentContextNode` `{ parentId, rootConversationId, edgeKind, folder, seed, status, returnSummary, stopReason, filesChanged, usage }`. **Status carries the honest outcome vocabulary** (`entities/Outcome.ts`): `running | ok | partial | error | killed` — `partial` = a usable deliverable that stopped early (budget / step-cap / breaker / stall-after-text; the old vocabulary silently recorded these as `ok`), `killed` = interrupted or stalled empty; `stopReason` is the typed WHY (`budget`/`step-cap`/`stall`/`interrupt(by: human|parent|shutdown|deadline)`/`provider`/`error`). **Every run funnels through ONE terminal path — `finalizeRun`** (`usecases/finalizeRun.ts`): recordReturn (terminal-once at the store: `WHERE status='running'`) → `bus.complete` → the `subagent_end` event, idempotent + infallible, emitted on EVERY exit shape — an abnormal exit can no longer skip the terminal event (the old finalize did, which is why dead agents looked alive in the UI). Spawned runs are **supervised fibers** (`bus.forkSupervised`, no more `forkDaemon(...catchAll(void))`); `bus.shutdown()` (wired into TUI/print/json/rpc/daemon teardown) interrupts + AWAITS them so a process exit records `killed(shutdown)` instead of stranding rows. The node's full message history persists; the `seed` column is a small **descriptor** (`task` | `selection` | `handoff`) — actual seed messages live only in `context_messages`.
- **Resume / branch / handoff**: `seedFromNode` + `seedMode: "resume"` appends the task to the existing node and continues in its context (same node grows — full history every turn); `"branch"` spawns a **child** node (`edgeKind: "branched"`) seeded from the source node's verbatim messages; `"handoff"` (prompted as the PREFERRED follow-up mode) spawns a child seeded with a **generated brief** of the source's work (`generateHandoffBrief` → `handoffToMessage`, seed kind `handoff`) — continuity at a fraction of the tokens. The routing policy lives in the tool description + `# Sub-agents` prompt section: one agent = one piece of work, fresh spawn by default, reuse only for direct follow-ups, cheapest sufficient seed. **Context-passing**: the spawner is the sub-agent's ONLY bridge (it sees only its `task` + folder + SCOPE.md + project knowledge — not the conversation, the user's goal, the plan, or siblings' findings), so the prompts demand a full brief in `task` (OBJECTIVE / CONTEXT / OUTPUT / BOUNDARIES, Anthropic's multi-agent protocol). As a structural backstop, every fresh/handoff spawn is seeded with a one-line **mission** preamble from `RunContext.mission` (the human's request, inherited down the subtree) so a leaf never works fully blind, and spawned workers are nudged (`core/prompts/sections.ts`) to **read the blackboard FIRST** — a sibling may have already posted a finding/contract that changes the work. **Model**: a spawn picks a model TIER via `role` (`general` default, `code` for writing code) or the agent definition's `role:` frontmatter — never a specific model (the old per-agent `model` footgun is gone; see Models & providers). The tree records how context branched and evolved.
- **Identity & bounds**: `RunContextRef` (a `FiberRef` in `usecases/runContext.ts`) carries `{ rootConversationId, parentNodeId, depth, tokenPool }` ambiently — `runAgent` seeds it via `Effect.locally`, each spawn re-seeds for its children. A `maxDepth` guard (default 2 — the root agent can spawn sub-agents, and those can spawn one more level) returns `MaxDepthReached` as a tool failure; each sub-agent run is bounded by its own step cap (default 80, `Settings.subAgentMaxSteps`, `:set subAgentMaxSteps <n>`, threaded live through `RunContext` like the pool). A capped run's summary is stamped `[stopped early: the step limit was reached — this result is partial]` (mirroring the budget stop), so the parent reads truncation instead of trusting a mid-thought sentence as the deliverable.
- **Token budget** (`usecases/tokenBudget.ts`): depth bounds termination, the **shared pool** bounds spend — one `Ref` (default **4M** billed tokens, `Settings.subAgentTokenBudget`, `:set subAgentTokenBudget <n>`, 0 = off) drawn from by every sub-agent in a turn's subtree. A drained pool refuses new spawns (`BudgetExhausted`, model-readable) and stops running sub-agents at their next turn boundary, marking the summary `[stopped early …]` so partial results read as partial. `:tree` rows show each node's billed tokens.
- **Web-lookup budget** (the convergence brake): the shared token pool is too coarse to stop a researcher that *over-fetches* (a 2-item question once burned 69 `web_fetch`es without converging). Each spawned agent gets a **per-run `web_fetch` + `search_web` counter** (`Settings.subAgentFetchBudget`, default 15, 0 = off; threaded `RunContext` → `ScopeBinding.fetchBudget` → `makeCodingHandlers`); past the cap the web tools refuse with a model-readable "report now from what you have", which `failureMode:return` turns into a graceful in-turn signal. The **root coder is exempt** (its binding leaves `fetchBudget` unset). The research-coordinator prompt also right-sizes the *fan-out* (a small question = ONE researcher), and the deterministic fleet-efficiency gate (see Self-improving loop) records an over-research lesson for next time.
- **Parallel fan-out & write-sequencing**: spawning is non-blocking, so multiple `run_agent` calls in one assistant turn each fork-and-return at once — the whole fleet runs concurrently (a turn's tool calls themselves resolve with `generateText`'s `concurrency`, default 4 — `DEFAULT_TOOL_CONCURRENCY` in `agentLoop.ts`). **There is no folder lock.** Read-only work (research / investigation / review) fans out in parallel — readers never conflict; **writing is sequenced by the ORCHESTRATOR**, not a lock: the coordinator protocol is *investigate in parallel → implement one writer at a time* (spawn a coder, `wait_for_agents`, spawn the next). The old per-folder semaphore held for an agent's whole run, so a hung writer stranded its same-folder siblings invisibly + unrecoverably (it's what wedged a live fleet of three read-only reviewers); sequencing at the orchestrator keeps control where it's observable (`wait_for_agents` timeouts) and recoverable. Because the fleet are background daemons (they outlive the spawning turn), teardown is explicit: **Esc** interrupts the root fiber AND `bus.interruptAll()`, and a runtime finalizer does the same on exit — structured cancellation, no orphans. **Wedged/dead nodes are handled too**: the spawn body wraps in `Effect.onExit` (`buildScopeRuntime.ts`) so a run that exits *without* recording a terminal return (interrupted / died) records an `error` return and notifies the parent immediately — the DB node never strands as `running`. A complementary **mid-session sweeper** (`inProcess.ts`, ~120s grace via `SWEEP_GRACE_MS`, `isNodeStranded`) flips any still-`running` node that's no longer live on the bus to `error` and notifies the parent, closing the gap the startup-only reconcile left. The Activity pane attributes parallel runs correctly: inner tool/assistant events carry `subAgentNodeId` (stamped by `makeInnerHooks`), `subagent_start` carries `parentNodeId`, and the pump keys each run to its own tree container (`onSubAgentStartKeyed`/`onToolStartUnder` in `executionTree.ts`) instead of the old single-open-stack model that nested parallel runs into a staircase. The events flow through the pump even for background daemons, so `:tree`/activity/the fleet chip update live after the root turn ends.
- **Staleness** (`usecases/staleness.ts`): a persisted context is a cache of a world that keeps changing. Nodes are stamped with the workspace git HEAD at finish (`workspaceRef`); resume/branch against a moved HEAD prepends a **staleness brief** (ref range + `git diff --stat` scoped to the node's folder + "re-read before editing") and `:tree` shows a yellow `stale` badge. All best-effort: non-git workspaces never stamp, a GC'd ref never breaks a spawn.
- **`SCOPE.md` is ambient folder context, not an agent definition**: `getScopePromptBody(folder)` (`discoverScopeTree.ts`) reads `folder/SCOPE.md` (frontmatter stripped) and injects the body into any sub-agent scoped there. Frontmatter `name`/`description` are inert metadata now; the old `delegate_to_<name>` tools are gone.
- **Events**: `onSubAgentStart/End` carry an optional `nodeId`, bridging the ephemeral per-run execution tree and the persistent context tree. In the TUI rail, a `run_agent` call renders as the sub-agent container itself (Task-style), not a redundant tool pill.
- **TUI** — the right pane is ONE always-visible **fleet tree** (`:tree`, `FleetTree` → `ContextTreeView` + a reactive **detail section** below, `view/panes/side/NodeDetail.tsx`, following the cursor — full unclipped summary/seed/files/billed tokens, a running node's **live tool feed** via `TreeNode.nodeId` + `findByNodeId`). It is **current-session-only** and **always expanded**: a single **non-foldable root** — the active session (its orchestrator shows a live `●` while its own turn runs) — with its coordinator/agents/sub-agents nested beneath as a **git-graph-railed tree** (`├─`/`└─` + `│` continuation, computed purely in `buildNavRows`; a `branched` connector is accent-tinted), each node a status glyph (`●`/`✓`/`✗`), provenance (`spawned`/`branched`/`resumed`), seed kind, files count, return summary, billed tokens. Loaders (`loadAgentTree` / `loadSessions` / `refreshNav` in `actions/contextTree.ts`, **`activeOnly` by default**) seed it at boot and refresh each turn end + on every sub-agent start/end — so the fleet appears and flips status live (the in-process `code` driver AND the remote `efferent` daemon path, which reads the shared store via `refreshNav` in `remoteRuntime.ts`). Other sessions are reached via the **`:browse`/resume** picker, not this pane — **no four-view cycle, no `sessions`/`fleets` tabs** (that model is gone). **`↵`** opens the row: a conversation **becomes the active session** (`switchToConversation` — resume + reload, refused mid-turn); an agent node opens a read-only **session preview** overlaying the conversation pane (`NodePreview` in `state/conversation.ts` — `viewBlocks = preview ?? blocks`, so the event pump keeps writing to the live rail underneath; title `agent: <folder>`, `── seed … ──`/`── run starts ──` boundary rules from `seedMessageCount`; `↵`-again/`q`/idle-`Esc` close). **While a preview is open the composer routes to that agent — like pairing with a teammate** (input title `input → agent: <folder>`), and it never freezes: if the node is **running**, the message is delivered to its **bus mailbox** (`bus.post` — it reads it at its next step, the composer stays free, no busy flip); if it's **finished**, the message **resumes the node in place** via `ScopeRuntime.resumeNode` (the human-driven `seedMode:"resume"` — staleness brief, depth-0 fresh budget, children hang off the node; refused while another turn runs). The preview re-fetches at turn end (`{focus:false}` refresh keeps the composer). **Routing is explicit: with a preview open the composer pairs with that agent (above); with no preview open, a message typed while the root is busy QUEUES (agy-style — a pending `▸` entry that runs as the next turn when the current finishes), NOT injected mid-turn.** The in-process `code` driver queues locally (`submit.ts` → `run.enqueue`, drained in `finishTurn`); the remote `efferent` driver queues on the daemon (`inProcess.send` → run-state queue, surfaced in `SessionState.queue`, mirrored into the pending list by `remoteRuntime`'s `refreshQueue`). Every user line — optimistic, queued-then-drained, or another client's — now arrives as a **`user_message` event** the daemon emits from `runAgent` (carrying the message's store position); the rail's user line rides the same keyed event pump as everything else (the old queue-diff reconstruction is gone). (Sub-agent *results* still auto-deliver to the root via its mailbox + `onTransformContext` — a separate path.) **`c`** forks an agent node's full context into a new conversation and makes it active (the human-driven branch — `continueFromNode`); `d` drops a node + descendants (nodes only, running guarded); refreshed when a run finishes spawning (`presentation/contextTreeView.ts`, `presentation/nodePreview.ts`, `view/panes/side/ContextTree.tsx`, `actions/contextTree.ts`, `actions/submit.ts:submitToNode`). Switching focus (Tab / Ctrl-hjkl) never reorders the panes.

## Conversation rail = a keyed cache (single source of truth)

The TUI rail (`state/conversation.ts` `blocks`) is a **normalized, identity-keyed cache** — the React-Query model: every writer (the live event pump, the optimistic submit, a DB re-projection on resume/attach/resync) addresses a block by **stable identity**, so the same logical message can't appear twice no matter how many writers touch it (the old duplicate-messages bug). Identity is the message's **absolute store position**: `append` returns it; `messageKey(position, "u"|"a"|"r", ord)` (`presentation/conversation.ts`) is computed **identically** by the live pump (off the event's `position`) and by `projectHistory` (off `baseOffset + index`, `baseOffset` = the window's absolute start, so it's **handoff-safe** — an absolute position never shifts when `listActive` narrows). `pushBlock` upserts by identity (a replay/re-projection lands on the same entry); `reconcile` (the reconnect-resync write) merges by key and keeps any live-but-unpersisted tail (never drops the streaming reply); `setBlocks` stays a blind replace for true document swaps (`:clear`/build). The user line flows through a **`user_message` event** the daemon emits from `runAgent` (carrying `position`); an optimistic line (`pushOptimisticUser` → `opt:<n>` + a FIFO of pending keys) reconciles onto its authoritative position by `resolveOptimisticUser` — never by content-hash (two identical-text messages stay distinct). The remote reconnect-resync (`remoteRuntime.ts`) is **idle-gated** (tail-only while a turn streams) with a strictly-exclusive cursor, so a blip can't drop or double the rail. Contract proof: `state/conversation.test.ts` (idempotent upsert, optimistic reconcile, reconcile-preserves-tail, cross-writer key equality, handoff-safety).

**Daemon-authoritative phase**: `SessionState` now carries a coarse lifecycle `phase` (`idle`/`thinking`/`tool`) derived by a **pure core reducer** (`reducePhase` in `core/usecases/agentPhase.ts`) the daemon folds every published *root* event through (sub-agent inner events + `run_agent` are skipped — they're the orthogonal fleet). The client reconciles its richer `agentState` machine against `getState(id).phase` on (re)attach + resync — so a client joining mid-turn shows `thinking` and an idle daemon clears a stale spinner, fixing the phantom-"thinking"-on-reattach bug. The CLI's `agentState.ts` reuses the same `AgentPhase` literal + transition rules verbatim, so the two never drift. **Tool-pill identity is deterministic**: `ensureToolCallIds` (`promptMapping.ts`) mints a stable id (`<turnIndex>:<toolName>:<ordinal>`) at the loop source for providers (notably Gemini) that omit tool-call ids, writing it into the response BEFORE it splits into events + the persisted `AgentMessage` — so live key === projected key and re-projection/re-attach never duplicates or reorders a pill.

## Handoff & conversation context

A **handoff** replaces the *loaded* history with a model-generated summary while keeping the originals. It's how a long session frees context without losing the record.

- **Storage** (`checkpoints` table, migration `0005`): a checkpoint is `{ messagePosition, summary }` for a conversation. Original `messages` rows are never modified. `ConversationStore` exposes `checkpoint(id, summary)` (atomic — folds at `COALESCE(MAX(position), -1)` in one statement), `getLatestCheckpoint`, `listCheckpoints`, `list` (ALL messages, for browsing), and `listActive` (only real rows **after** the latest fold).
- **Loading** (`runAgent`): loads `getLatestCheckpoint` + `listActive`, prepending the summary as a single synthetic `user` message via `handoffToMessage` (`promptMapping.ts`) — domain logic stays in core, not the adapter. So the model sees `[handoff summary] + [messages since the fold] + [new user msg]`; everything before the fold is preserved but never re-fed.
- **Creating** (`createHandoff`, `@xandreed/sdk-core/usecases/handoff.ts`): summarizes the **currently loaded view** (prior summary + active window, never the raw `list`) with `HANDOFF_PROMPT` (`prompts/handoff.ts`, a Goal/State/Next-steps/Constraints brief) and writes a checkpoint. Summarizing the loaded view keeps handoffs **cumulative** — a second handoff folds the first summary back in. No-op when nothing new since the last fold.
- **TUI**: `:handoff` runs it (pushes a checkpoint block); `:context` toggles the **context viewer** in the side pane (`cli/presentation/contextView.ts` — `buildContextView` partitions `list` + `listCheckpoints` into *archived* segments (folded, not loaded) and the *loaded* segment; `buildContextRows` groups each segment's messages into **foldable, selectable turns**, rendered as a tree so the replacement is visible); `:browse` lists workspace conversations and `:resume <#|id>` switches to one (full replay for browsing, active-window for execution). On boot with no `--resume`, if the workspace has prior conversations, a **startup picker** (`openConversationPicker` → a select overlay, purpose `conversation`) floats "Resume a conversation" over the fresh session — a leading "＋ Start a new conversation" row then `<date> · <title|first-prompt>` per conversation (the title is generated on the utility model after a session's first exchange — `generateSessionTitle` → `ConversationStore.setTitle`); Enter resumes via `resumeConversation`, Esc / "start new" leaves the fresh session. No prior conversations → straight to an empty rail.
- **Curate context → new session**: in the context viewer, `Space` selects/deselects a **turn** *or* a **handoff** (the `⚑` archived-segment header) and `b` (or `:build`) **builds a new conversation** seeded with only the picked units, switching to it (`messagesForSelectedTurns` → `ConversationStore.create`/`append` → the `:resume` switch). A selected handoff contributes **only its summary, as one `handoffToMessage` message** (not the folded originals); a handoff and its own inner turns are therefore **mutually exclusive** (selecting one clears the other — `archivedTurnRanges`/`handoffOwningTurn` in `contextView.ts`). Turn granularity keeps tool-call/result pairs valid; the original conversation is untouched.

## Context compaction (compression without breaking cache)

Cache-safe context compression, implemented natively in `core/usecases/compaction.ts`. The governing constraint: provider prompt caches key on a **byte-stable prefix**, so compression must never rewrite history. Four tactics:

1. **Append-time tool-result compression** (`compressToolResults`, called by `runAgentLoop` on each step's new tail): any string inside a tool-result output over the budget (`Settings.toolResultMaxTokens`, default 4000 tokens ≈ 16k chars, 0 = off; threaded to sub-agent loops via `RunContext` like the step cap) is compressed the moment it enters the buffer — the persisted record and every future prompt carry the compressed form from byte one, and nothing already sent is ever rewritten. Compression is **structure-aware first** (`compactionContent.ts`, the SearchCompressor/LogCompressor ideas from compaction ported to our shapes): grep-shaped output (`path:NN:` lines, any tool) is grouped per file — every file visible, first 5 matches each, exact counts in the marker; Bash output with error signal keeps head/tail + every error with its trace block intact (a state machine absorbs `at `/`File "`/indented continuations) + deduped warnings (`(×N)`) + test-runner summary lines, re-emitted in original order with inline `[…N lines omitted…]` gap markers. Shapeless text falls back to the blind head (75%) + tail (12.5%) clip. **The TUI still shows the full output** (hooks emit the raw response before the tail is compressed).
2. **Reversible markers** — the clip marker names what was dropped (`[…compaction: ~4509 tokens of this Bash output omitted…]`) and how to get it back (re-read with offset/limit, narrower grep, `| head`) — compression the model can undo on demand. Live-verified: the model greps its way to the buried line instead of trusting the hole.
3. **FAST-tier middle digests** — when `UtilityLlm` is in context and the dropped middle is big enough, the fast role writes a ≤120-word summary into the marker (the fast tier's first consumer). Usage is reported via the new `onHelperUsage` hook → `helper_usage` event → `byRole.fast`. Best-effort: no service / provider error degrades to the plain marker; sub-agent loops forward the spend to the parent ledger.
4. **Threshold auto-fold** (`autoHandoffPct`, default 85, 0 = off): when a finished turn's context crossed the threshold, the TUI runs `:handoff` automatically at the boundary — one deliberate prefix rebuild (the only cache-safe way to shrink history), then the cache is warm again. Skipped while messages are queued and on `~` resume estimates.

Note: `read_file`/`web_fetch` carry their own per-tool caps (`truncated: true`); compaction is the *generic backstop* for anything without one — bash stdout, grep floods, future tools. Verified live: an 80k-char `cat` persisted at 14.6k chars with marker + digest, `Σ` showing all three tiers.

## CLI shape

Composition root: `packages/cli/src/main.ts`.

**Subcommands (the run-paths are first-class commands).** The CLI runs agents three ways, and the split is a subcommand, not a flag:

- **`efferent`** (no subcommand) — the **default master TUI**: attach-or-spawn the per-workspace daemon and drive it (the split-process path). `EFFERENT_LOCAL=1` forces the in-process driver for this default invocation; `EFFERENT_REMOTE` is the explicit remote alias.
- **`efferent code`** — the **focused single-fleet coder, IN-PROCESS** (in-memory Workspace, `variant: "code"`) — the bundled coding agent run without a daemon (replaces the old `code` bin / `--code` flag, both deleted along with the `dist/code.js` shim).
- **`efferent attach`** — explicitly attach the master TUI to the daemon (auto-spawn if absent).
- **`efferent daemon start`** (alias **`serve`**; was `--mode daemon-serve`) — run the persistent per-workspace HTTP/SSE daemon thin clients attach to. **`efferent daemon status`** / **`efferent daemon stop`** manage its lifecycle.
- **`efferent verify`** — the **graded acceptance battery** (`packages/cli/src/verify/`): runs against a `--target` (`source` default · `commit:<sha>` worktree · `release:<ver>`/`npm:<spec>` clean-room Docker) in three tiers — **A** deterministic/no-key (boot · the no-provider gate · the **internal UI flows** via `testRender` bun tests in `cli/verify-flows/` · daemon lifecycle), **B** LLM-as-agent objective (real keyed turns through in-process/daemon/rpc, asserting a *file side-effect* not prose, on `opencode:deepseek-v4-flash` by default; the daemon turn uses `createFleet` so the root coder writes the file), **C** LLM-as-judge semantic (a cheap eval subset, soft unless `--strict`). Reuses `makeHttpTransport`/`probeHealth` for the daemon, the evals `--json` for Tier C, and the in-repo `test/docker/run.sh` for the container target. Exits non-zero only on a hard `fail` (skips/softs don't). `--tier A|B|C|all`, `--model`, `--strict`, `--json`, `--keep`. `verify`/`eval` join `code`/`attach`/`daemon` as reserved first-token words.
- **`efferent eval [suites…]`** — first-class access to the eval suites (forwards to `packages/evals/src/run.ts`; source checkout only). Flags `--main`/`--fast`/`--judge`/`--samples`/`--config`/`--json`.
- **`efferent "<prompt>"`** (print), **`--mode json|rpc|daemon`** — the headless modes, unchanged.

Modes under `packages/cli/src/modes/` (`--mode <x>`, default `auto`):

## Models & providers (multi-provider router)

The agent loop talks to one provider-agnostic `LanguageModel`; which provider/model backs it is a **runtime selection**, not a compile-time layer choice.

- **Router** (`adapters/src/llm/router.ts`, `RouterLanguageModelLive`): a `LanguageModel` whose `generateText`/`streamText`/`generateObject` read `ModelRegistry.current` on every call, resolve a key from the `AuthStore` (refreshing an expired OAuth token first), and build the chosen provider's `@effect/ai` service **per request** from that key over a shared `FetchHttpClient` (`Effect.scoped`/`unwrapScoped` so the client lives exactly one call). So a credential added mid-session via `:login`, or a `/model` switch, takes effect on the next turn with no rebuild. There is no longer a `clients.ts` — keys aren't captured at layer-build.
- **Transient-failure resilience** (`retryableLlm` in `adapters/src/llm/retry.ts`): the default provider (Kimi via the opencode gateway) returns 429 "engine overloaded" under load, and with no retry that was instant turn death and a thrashing fleet. `retryableLlm` wraps each LLM *connect* (never a live stream) and retries only **transient** failures, decided by the shared **provider-defect taxonomy** (`classifyProviderError` in `core/usecases/` → `transient | quota | config | auth | model`); never a 4xx client error or a decode bug, which are permanent. Up to 3 retries, exponential **1s → 2s → 4s** (capped 8s) with ±25% jitter, honoring `Retry-After` when present (clamped ≤60s — longer classifies `quota`); each retry annotates the active span (`llm.retry.*`) + logs a warning. Every router call is also bounded by a **120s fiber-level request timeout** (`withLlmTimeout` — the official Google/OpenAI/Anthropic paths shipped none), classified transient so it retries. Wired at the **router** (`generateText`/`generateObject`, covering every provider incl. the Kimi/opencode gateway) plus `utilityLlm` + `webSearch`.
- **Role failover on persistent defects** (`withFailover` in `router.ts`): when a call dies with a **`quota`** (insufficient balance / daily-quota Retry-After / usage limit) or **`config`** (invalid-request 400, endpoint 404) defect, the router retries the call ONCE on the fallback selection — the **code** role falls back to the run's pinned **general** model; the **general** role to the human-configured **`Settings.fallbackModel`** (unset ⇒ no failover, the error surfaces). Never a model the agent chose; never the selection that just failed. Loud by construction: the notice rides the retry sink (rail / node log / health suffix), the span gets `llm.failover.*`, and a `[failover: kimi → glm after quota]` note is folded into the run's terminal outcome (`RunContext.failoverNotes` → `RunOutcome.notes`). `auth` (401/403) never fails over — it surfaces with the `:login` hint; `model` (MalformedOutput) belongs to the loop's own corrective recovery.
- **Anthropic OAuth** (subscription): when the credential is `oauth`, the client is built with `apiKey: undefined` + a `transformClient` that sends `Authorization: Bearer …` and `anthropic-beta: claude-code-20250219,oauth-2025-04-20` (no `x-api-key`), and the router prepends the required `"You are Claude Code, Anthropic's official CLI for Claude."` system block (`Prompt.merge`). Protocol + constants in `adapters/src/auth/oauth/anthropic.ts` (PKCE, authorize URL, exchange, refresh); callback server + browser-open in `cli/src/login/oauthServer.ts`.
- **Selection** lives in `SettingsStore` as `settings.model = "<provider>:<modelId>"` (the single source of truth, persisted to `.efferent/config.json`). `parseModel`/`formatModel`/`contextWindowFor`/`defaultModelForProvider(s)` are pure helpers in `@xandreed/sdk-core/entities/Model.ts`.
- **`ModelRegistry`** port (`@xandreed/sdk-core/ports/ModelRegistry.ts`): `current` (parsed selection), `list` (live catalogue), `select` (persist + return). `ModelRegistryLive` (`adapters/src/llm/modelRegistry.ts`) fetches the catalogue over **raw HTTP** (Google `…/v1beta/models`, OpenAI `…/v1/models`, Anthropic `…/v1/models`) and parses defensively; keys come from the `AuthStore`, so only logged-in providers are queried (Anthropic OAuth uses a Bearer header).
- **`ModelLive`** bundles router + `ModelRegistry` + dynamic `LlmInfo` over an internal `FetchHttpClient`; requires `SettingsStore` + `AuthStore`. Replaces the old single-provider `GoogleLive`.

**Model roles (general / code / fast)** — one mental model, three knobs (`ModelRole` + `AgentModelRole` + `modelForRole`/`roleIsConfigured`/`selectionFromString` in `entities/Model.ts`). A running agent NEVER picks its own model — its role is structural, and the human owns which model backs each tier (the Aider `--model`/`--editor-model`/`--weak-model` split). The three roles:

- **general** — the root conversation / orchestrator AND research / analysis / planning sub-agents: the general-purpose brain (`settings.model`, the `/model` switch).
- **code** — the coding sub-agents (the fleet writing code) (`settings.codeModel`; unset → general). A sub-agent that writes code runs here; one that researches runs on general. The **spawner picks the tier** (`run_agent({ role: "general" | "code" })` or an agent definition's `role:`), never an arbitrary model — default `general`.
- **fast** — one-shot helper calls off the loop (`settings.fastModel`; unset → general): compaction middle digests (see Context compaction), auto-approval judgments (see Agent loop → Bash safety), and **session titles**. Helper-only — never a sub-agent's role. Reached through `UtilityLlm.complete(prompt, { role: "fast" })`.

The run **freezes all three role models at start** into `RunContext.pinnedModels`; the router resolves a fiber's call to `pinnedModels[role]` (the fiber's `RunContext.modelRole`, set to its spawned role), so a mid-run `/model` / `:set codeModel` can't move a running fleet — which also keeps each provider's prompt-cache prefix warm. `UtilityLlmLive` returns `{ text, usage }` so each tier's spend is countable; per-role billed tokens accumulate in `SessionStats.byRole` (root → general; a sub-agent → its role, carried on the `assistant_message` event's `subAgentRole`; helper usage → fast).

**`/model`** (TUI): no arg opens the live-catalogue picker for **general** (persists, updates the status bar); **`:model code`** / **`:model fast`** open the same picker for that role, with a leading *default (follow general)* row to clear it. Direct path: `:set codeModel`/`:set fastModel <provider>:<modelId>|default`. Switching provider mid-conversation applies going forward and surfaces a one-line hint (Gemini 400s on prior non-Gemini tool-calls that lack a `thought_signature` — `/reset` if it errors).

**Caching** is aggressive but provider-native, and the governing rule is one line: **a byte-stable prefix per agent**. Each agent keeps its OWN cache — there's no cross-sub-agent carryover, so a sub-agent's distinct system prompt + `code`-tier model is simply its own prefix, and the `pinnedModels` freeze (above) means neither a `/model` switch nor a `:set` moves a *running* agent's model and invalidates its prefix mid-flight. The seed is stable too: the mission preamble + task are written once at spawn, never rewritten. Per provider: OpenAI gets automatic prompt-prefix caching + a stable `prompt_cache_key`; Gemini relies on implicit context caching (stable prefix → `cachedContentTokenCount`); **OpenCode/Kimi** (the default provider) auto-caches a stable prefix server-side, and `openCode.ts` reads the hit back (`cached_tokens` / `prompt_cache_hit_tokens`, six vendor fallbacks) so the status bar `cache%` lights up with no extra config; **Anthropic requires explicit opt-in** — the router stamps `cache_control: ephemeral` breakpoints (`withAnthropicCacheBreakpoints` in `providers.ts`) on the last system message + the last two non-system messages every call (markers move with the tail; placement isn't part of Anthropic's content hash, so old prefixes keep hitting). Helper tiers (UtilityLlm, web search) deliberately don't stamp — one-shot prompts would pay the +25% write premium for nothing. Anthropic's `input_tokens` EXCLUDES cache reads/writes, so `extractUsage` folds the raw anthropic metadata back in (else the gauge reads ~0 on a cached turn). Live-verified: turn 2 on Sonnet = `93% cached`. Explicit Gemini `cachedContent` is not expressible through `@effect/ai-google` today (it always sends full `contents`), so we don't fake it.

A bare per-provider substring heuristic remains as the fallback for ids newer than the last snapshot; dated ids (`…-20260101`) fall back to their base id. ## Skills

`.efferent/skills/*.md` files are auto-discovered at startup. The search path walks `cwd → parents → ~/.efferent/skills/`; closer-to-cwd shadows farther on name collisions. Each skill file has YAML-ish frontmatter and a free-form markdown body:

```
---
name: <slug>
description: <one-line summary for the prompt>
---

(detailed procedure for the agent to follow)
```

At startup, names + descriptions are injected into the coder system prompt under a `# Skills` section. The bodies are lazy-loaded by the model via `read_skill({ name })` only when relevant — reusable procedures you can ship without changing the code.

Loader: `loadSkills(cwd, homeDir)` in `@xandreed/sdk-core/usecases/loadSkills.ts`. Failures (missing dirs, malformed frontmatter) are silently skipped — a broken skill never breaks the agent.

## Self-improving loop

The compounding half of the runtime: each finished task can leave behind a **verified, reusable artifact** the next run inherits, so the agent that ran a task yesterday is smarter today. It runs in **two automatic places** (both default-on, `:set autoLoop`/`autoDistill off` to disable), and Opus only ever **validates** — the Kimi swarm does the work. Full design + literature grounding (ACE / AWM / SkillOps) in `docs/self-improving-loop.md`.

**(A) Online — the swarm task loop (validate → learn → retry).** When the fleet handles a substantial task, the **coordinator** (`usecases/teamAgents.ts`) runs the Kimi swarm, the Kimi **architect** reviews each piece (cheap, continuous), and then the coordinator submits the whole deliverable to the **independent Opus gate**: `verify_with_gate({ task, summary, files })` → `Verifier.gate` (`ClaudeHeadlessVerifierLive`, `claude -p` in the repo dir, reads the real files). On `needs_work` the coordinator **learns** (`note_constraint({ rule })` → a `.efferent/CONSTRAINTS.md` bullet, persisted + auto-loaded) and **retries** the failed pieces with the gate's reasons in the brief — looping until `sound` or the cap (`maxLoopAttempts`, default 3; token budget + depth are the hard ceilings). **Fail-soft**: a missing `claude` / gate error → the coordinator falls back to the architect verdict, never blocks the task. The independent gate is what makes a prompt-driven retry trustworthy — the swarm can't "good-enough" its way out (live-verified: Opus catches a deliverable whose summary lies about the code). The new tools live in the scope-runtime toolkit (`buildScopeRuntime.ts` `loopToolEntries`); `Verifier` is threaded through the handler layer (provided by `AppLive`; evals/tests use `UnavailableVerifierLive`). **This is now also enforced structurally — see (E):** the same Opus gate runs in `driveLoop` after ANY sub-agent run (fail-closed), so verification no longer depends on a `coordinator` being in the roster or on the model choosing to call `verify_with_gate`. The coordinator's `verify_with_gate` stays an *in-fleet* checkpoint (fail-soft, falls back to the architect); the `driveLoop` gate is the *mandatory* end-of-objective one.

**(B) Offline + at turn-end — the distiller.** Three roles, and the merge is deterministic: **Reflector** (`distill()` / `runDistillation()`, `core/usecases/distill.ts`) mines candidate learnings from a conversation on the **fast** tier (cheap/Kimi) — abstracted routines (AWM), success-gated for skills, failure-gated for constraints; the **Verifier** gate (`Verifier` port + `ClaudeHeadlessVerifierLive` in `adapters/src/verifier/`) refutes each candidate by running **Opus via the real `claude` headless CLI in a CLEAN-ROOM sandbox** (the sanctioned subscription-rate path + an independent grader that checks against ground truth) — **fail-closed**, an unverifiable candidate is dropped. **Clean-room** (so the gate is a genuinely independent referee, not one steeped in the project it judges): claude runs in a fresh temp dir under a CONTROLLED validator `CLAUDE.md` (NOT the project's `AGENT.md`), with a **pinned model**, reaching the repo READ-ONLY via `--add-dir` ONLY when the check is code-related (a `project`-scoped learning, or a deliverable with file changes) — a general rule (`use const`) or a prose/research deliverable is judged on its own merits, no repo context. **Airtight:** `HOME` points at the sandbox with ONLY claude's credentials (`~/.claude/.credentials.json`) copied in — it authenticates but loads NO global `CLAUDE.md`/memory/config (verified: inherited context drops to ~nothing); if the creds aren't a file (keychain) it falls back to HOME-intact (sandbox cwd still excludes the project narrative). the **Curator** (`persistArtifact()`, `core/usecases/persistArtifact.ts`) merges survivors as **delta items** (never a wholesale LLM rewrite — the ACE context-collapse rule) into `.efferent/skills/<slug>.md` (the first programmatic skill write), `.efferent/memory/<slug>.md`, or a bullet in `.efferent/CONSTRAINTS.md`. Constraints auto-load as a distinct `# Constraints` prompt section via the instruction-file channel (`discoverInstructionFiles.ts`, `kind: "constraints"`). It also fires **automatically at each turn boundary** (`autoDistill`, default on) — the TUI forks a background `runDistillation` over the just-finished conversation (mirrors the auto-handoff fire in `cli/actions/submit.ts`), pushing a `learned N lessons` rail line; and via the CLI **`efferent distill [--since <date>|--conversation <id>|--dry-run|--limit <n>|--threshold <0-1>]`** for backlog mining (`--dry-run` is miner-only, no `claude`, no writes — everything's in `ConversationStore`). Config: `EFFERENT_CLAUDE_BIN` (default `claude`), `EFFERENT_VERIFY_MODEL` (default **pinned** `claude-opus-4-8`; set `opus` if a `claude` install doesn't know the id), `EFFERENT_VERIFY_ARGS` (default `--permission-mode plan`, read-only). Settings: `autoLoop` / `autoDistill` (default on), `maxLoopAttempts` (default 3). The mandatory end-of-objective gate (see (E)) emits a dedicated **`gate` `AgentEvent`** (`sound`/`needs_work`/`blocked`/`unavailable`), rendered as a rail line by the TUI pump. Designed-not-built: a daemon-integrated scheduled distill, the helpful/harmful counter feedback, embedding dedup/maintenance, and a dedicated `distill_*` TUI event (the learn step still surfaces as the `learned N lessons` line + `note_constraint` pills). See the doc.

**(C) Convergence + closing the loop everywhere (2026-06-27, full design in the doc).** Three gaps made the fleet a *one-off* in practice; all four closed:
- **The loop fires on EVERY run path now**, not just `efferent code`. `runAutoDistill` (`core/usecases/autoDistill.ts`) is the shared turn-boundary step, fired from the **daemon** (`inProcess.finishTurn` — it never distilled before), **headless print/json** (`modes/headlessDistill.ts`, awaited + bounded, gated on `autoDistill`), and the TUI submit (refactored onto it). A `learned` `AgentEvent` surfaces persisted lessons on every path.
- **The over-research lesson now COMPOUNDS.** `runAutoDistill` also runs a **deterministic fleet-efficiency gate** (`efficiencyGate.ts`) — reads the run's persisted context tree; if the fleet over-worked it (too many workers / too many billed tokens — egregious thresholds, so a right-sized run makes no noise) it persists a canonical `fleet-research-budget` constraint **with no LLM and no Opus gate** (trustworthy by construction). It catches the runaway the per-worker fetch cap can't: too MANY workers, each individually capped. The LLM distiller's miner + Opus refute prompts were also **broadened to admit process/efficiency lessons** (judged on whether following the rule improves the next run, not on whether an error occurred — previously a "converge faster" lesson fit no candidate kind and the gate rejected it).
- **The research fleet now has the Opus gate too.** `researchCoordinatorAgent(opts)` (`usecases/researchAgents.ts`) mirrors `coordinatorAgent`: `autoLoop` adds `verify_with_gate` + `note_constraint` and a VALIDATE→LEARN→RETRY phase before REPORT. `buildGatePrompt` **branches on whether files changed** — a research deliverable is PROSE (no files), so Opus judges the *answer* (addresses the task, well-sourced, honest, coherent) instead of reading changed files. (Before this, the research fleet had no validation step at all.)
- **Evals prove it:** the `research-efficiency` suite budgets fetches/tokens/spawns from the fleet-wide trajectory (the 69-fetch runaway scores ~0 where `swarm` scored it a perfect 1.0), and `cli/usecases/loopClosure.test.ts` proves the loop CLOSES deterministically (gate → `persistArtifact` → `discoverInstructionFiles` → `# Constraints` in the next run's prompt).

**(D) Scope, corrections, and meta-prompts (2026-06-27).** Every learning now carries **`scope`** (`global | project`) + **`source`** (`user | inferred`), miner-classified. **Scope routing:** `persistArtifact` writes a `global` learning under `~/.efferent/` (a general Effect/style/language rule, inherited by EVERY project — the read side already walks `cwd → parents → ~/.efferent`) and a `project` one under `<repo>/.efferent/`; the `globalRoot` arg is threaded as `homedir()` from submit/daemon/headless/distill-cli. **User corrections bypass the gate:** a rule you STATE (USER turns are tagged in the transcript; the miner always captures them as `source:"user"`) is persisted DIRECTLY — no Opus refutation (the human is the authority; same trustworthy-by-construction bypass the efficiency gate uses) — so you state "use `const`" / "no try/catch in the domain" once and never repeat it. **Meta-prompts:** a new `process` learning kind (HOW to work — plan first, check assumptions) is filed in a loadable **operating-guidance overlay** `.efferent/prompts/coder.md` (discovered by `discoverInstructionFiles`, rendered as a high-priority `# Operating guidance` section above `# Constraints`, editable by hand). The loop can edit it — but a `process` learning **ALWAYS passes the Opus gate** (the user-bypass NEVER applies — changing the agent's own instructions is high-stakes) and the edit is a deterministic delta bullet (append/update by id, ACE-safe, no LLM rewrite). The built-in `coderPrompt` stays the floor.

**(E) The gate is mandatory now — every swarm objective, not just the coordinator's (2026-06-28).** Before this, the Opus deliverable gate only ran if the model both delegated through a `coordinator`/`research-coordinator` AND called `verify_with_gate` — and it was fail-soft. So a root that fanned out to plain sub-agents (or did the work itself) shipped **unverified**. The gate now runs at the **structural chokepoint**: `driveLoop` (`usecases/runAgent.ts`), the one path `runAgent`/`resumeAgent` — and therefore every mode (TUI, daemon, print/json/rpc) — funnel through. After the loop returns, if the run created any sub-agent context-tree nodes (detected via `ContextTreeStore.listTree`, settled to terminal status), `driveLoop` submits the aggregate deliverable (`finalText` + the fleet's union `filesChanged`, judged against the run's `mission`) to `Verifier.gate` once. **Fail-closed, with the full loop:** `needs_work`/`blocked` → emit a `gate` event with the reasons → **LEARN** (`runAutoDistill` mines + Opus-verifies new skills/memories/constraints, gated on `autoDistill`) → **RUN AGAIN** (re-invoke the loop with the gate's reasons as a feedback message; persisted lessons auto-load for future runs) → **re-gate**, looping to `maxLoopAttempts` (default 3). A genuinely unavailable verifier (no `claude`) emits a loud `unavailable` `gate` event and proceeds — never a silent pass, never an infinite loop. Gated by the existing `autoLoop` knob (default on); it has no separate toggle and depends on neither a coordinator nor a tool call. Proven by `usecases/runAgent.test.ts` (no-subagents→no-gate · sound→gate-once · needs_work→retry→sound · cap · unavailable) and a live `--mode json` swarm (gate → needs_work → learn → retry → re-gate, with a real `gate` event).

**(F) ONE gate tier — the coordinator tier is deleted; the gate TOOLS stay gone (2026-07-02, supersedes the 2026-06-29 both-tiers design).** The 06-29 change made each lead (coordinator / research-coordinator) gate its own subtree structurally before returning. **Run forensics killed that design**: the lead-tier gate ran `settleChildren` polls plus a `claude -p` Opus subprocess (30-min cap) *inside the sub-agent stall watchdog's 180s no-progress window*, with no progress stamps — so finished leads were routinely interrupted mid-gate and recorded `[stalled]` with `filesChanged: []`, silently discarding completed, test-green work (the largest error bucket: 34 stalled + a share of 43 interrupted nodes). Gating is now **root-only**: `driveLoop`'s (E) aggregate pass is THE gate; `runSpawnedAgent` runs no gate, `isLead` is gone, and `gateOnce` (`core/usecases/gateLoop.ts`) is a pure decision requiring only `Verifier` (no in-gate distill — learning is single-shot at the turn boundary via `runAutoDistill`). The `verify_with_gate` / `note_constraint` tools remain removed (the model never drives the gate); the **architect** stays as the in-fleet, fine-grained per-piece review. The exit finalizer now also **preserves an interrupted/stalled run's produced work** (last assistant text + filesChanged + usage; a stall AFTER text ⇒ ok-with-note, never an empty `[stalled]` error), and fleet kills are **subtree-scoped** (`bus.interruptSubtree` — Esc and the headless deadline no longer `interruptAll` every fleet on the bus; the headless deadline default rose 6 → 20 min). Proven by `gateLoop.test.ts`, `buildScopeRuntime.test.ts` (a lead returns ok with ZERO verifier calls; a stalled run with text keeps it), and `agentBus.test.ts` (subtree kill leaves the other fleet alive).

## Evals

A fourth package, **`packages/evals`** — a minimal, Effect-native eval library: the `data → task → scorers` shape expressed as Effects, so a `task` can be the real agent loop and a `Scorer` can itself call an LLM. No sqlite, no UI, no persistence; runs under Bun directly.

- **Framework** (`src/framework/`): `EvalSpec<I,O,T,R>` is pure data (`defineEval`); `runEval(spec)` returns `Effect<EvalReport, never, R>` — every per-case/per-scorer failure is captured via `Effect.exit` (so a provider 429 scores 0, never crashes the run) and `R` is left open for the caller to provide once. Scorers (`scorers.ts`): `predicate`, `includesAll` (substring-coverage ratio), `fromEffect`, and `llmJudge` (LLM-as-judge, parses `{"score","reason"}`). `report.ts` prints a coloured per-suite table.
- **Env** (`src/env.ts`): `EvalEnvLive` mirrors `main.ts`'s composition but swaps Postgres for an in-memory `ConversationStore` (`support/inMemoryConversationStore.ts`, replicating the position/checkpoint fold semantics) so evals need no Docker. `support/workspace.ts` gives `withTempWorkspace` (acquire/release temp dirs); `support/coder.ts` (`runCoder`) stands up a real coder agent over a temp repo and reports the tools it called + final text + read-back files.
- **Config injection** (`src/config/`): a `RunConfig` pins `{main, fast, judge, promptVariant, maxSteps}`; `makeEvalEnv(config?)` swaps in a `FixedSettingsStoreLive` (its `load()` ignores disk) so a `--config matrix.json` run is the independent variable A/B'd against a baseline. No config ⇒ today's `LocalSettingsStoreLive`.
- **Suites** (`src/suites/*.eval.ts`): `handoff`, `tool-selection`, `coder-edit` (originals) + `whole-task` (bug-fix / multi-file / refactor / failing-test / read-only Q&A through the FULL loop), and three **fast-model** suites calling the fast-tier use cases directly — `judge-approval` (labelled allow/prompt accuracy), `compaction-digest` (`compressToolResults` fidelity), `session-title` (`generateSessionTitle` quality).
- **Run**: `bun run eval [name …] [--config f] [--main m] [--fast m] [--json]` (`src/run.ts`), or **`efferent eval [name …]`** (the same runner, fronted by the CLI). Gated on `hasKey` — no key → skip cleanly. Each run is collected **in-memory** and the report is **built from the spans** (`trace/process.ts` → `trace/report.ts`); set `OTEL_EXPORTER_OTLP_ENDPOINT` and it also streams to Grafana. Unit tests (`bun test`) cover the framework, the store fold contract, and the pure trace processor (synthetic spans), with no LLM/Docker.
- **`efferent verify`** (see CLI shape) is the **acceptance** layer above the evals: it drives the *real CLI* end to end (boot · UI flows · daemon · keyed turns) and folds a cheap eval subset in as its Tier C. Where evals score the agent's *quality*, verify proves the *build works*.

## Observability (traces + metrics)

Trace-first, by design: **production code only annotates** OpenTelemetry spans and records a few Effect `Metric`s — it never builds a metrics store. The waterfall is the full execution: `agent.run` (one per **user message** — carries `agent.conversation_id`, an `agent.prompt` excerpt, `agent.model`, run-total tokens) → `agent.turn` → `llm.generate` (model + `gen_ai.usage.*` + `gen_ai.cost_usd` + `gen_ai.cache_hit_ratio` + finish reason, plus the clipped `gen_ai.prompt`/`gen_ai.completion` text — captured whenever telemetry is on, no separate flag) → **`agent.tool`** (one per tool call — `agent.tool.name` / `.ok` / `.args_summary`, the seam in `recoverMalformedToolCalls`) and **`agent.subagent`** (a spawned run's subtree — node/depth/folder), plus the helper spans `agent.approval.judge` / `agent.compaction.digest` / `agent.title`. Metrics (`core/src/telemetry/metrics.ts`): `gen_ai_tokens_total`, `gen_ai_calls_total`, **`gen_ai_cost_usd_total`** (priced via the same `costUsd` catalogue the evals use), `agent_turns_total`, `agent_turn_latency_ms`, `agent_tool_calls_total`, **`agent_errors_total{kind=run|turn|tool|llm,error}`** (RED's E), `approval_verdicts_total`. **Bounded tags only** — conversation/node/run id + prompt live on **spans**, never on a metric. Everything is **inert without a tracer** — zero overhead un-exported.

- **Export** is one layer at the edge: `OtlpTelemetryLive` (`adapters/src/telemetry/otlp.ts`, Effect-native `Otlp.layerJson` over HTTP — no heavy OTel SDK in the CLI bundle). It's wired through a `Layer.unwrapEffect` in `main.ts` that reads the **`Settings.telemetry`** flag (the SOLE switch, `:set telemetry on`; schema default off) and returns `OtlpTelemetryLive` or `Layer.empty` — no env var enables export. Resource: `service.name=efferent` + `deployment.environment=production`; the endpoint defaults to `http://localhost:4318` (`OTEL_EXPORTER_OTLP_ENDPOINT` overrides WHERE, not WHETHER).
- **Eval / prod separation** — one stack, logical split. Evals export under `service.name=efferent-evals` (→ Prometheus `job`) + `deployment.environment=eval` + a `resource.eval.run_id`, into a **separate Grafana folder**; prod dashboards filter `{job="efferent"}`, eval dashboards `{job="efferent-evals"}`. Evals still **build their report FROM in-memory spans** (`trace/process.ts`) and only ship to the stack when `OTEL_EXPORTER_OTLP_ENDPOINT` is set.
- **Grafana dashboards** (`grafana/otel-lgtm`; auto-provisioned from `observability/grafana/dashboards/{production,evals}/` via two folder providers). Production: **fleet health** (`efferent-fleet` — RED + cost $ + cache-hit + tool success, SI/`$`/`%` units), **conversations** (`efferent-conversations` — Tempo table, one row per user-message trace, data-links into) **conversation** (`efferent-conversation?var-conversation=…` — that session's messages + tool calls → click a trace for the waterfall). Evals: **eval runs** (`efferent-evals` — per-suite mean/pass/cost + a `$run`-scoped case-trace table). The CLI deep-links out: **`:traces`** (active conversation) / **`:dashboard`** (fleet), Grafana base from `Settings.grafanaUrl` (default `http://localhost:3000`); `bun run eval` prints an eval-dashboard link when exporting. The required panels are Prometheus + Tempo-search; TraceQL-metrics panels are fail-soft enhancements. Full runbook + caveats in `observability/README.md`.
