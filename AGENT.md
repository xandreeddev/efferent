# efferent ŌĆö agent runtime on Effect.ts + Bun

An **agent runtime on Effect.ts** ŌĆö a pure-domain SDK (agent loop, context tree, compaction, multi-provider router, approval ŌĆö all composable Effects/Layers) plus **one CLI that runs agents** (an interactive TUI, headless print / json / rpc, or a persistent per-workspace daemon that thin clients attach to). The bundled coding agent is just one app the runtime runs (reached via `efferent code`). Built in public as `@xandreeddev`. See `../CLAUDE.md` (parent tree) for the broader project rules ŌĆö alias identity, OPSEC, weekly cadence, locked stack. See `docs/roadmap.md` for what's deferred and `docs/models.md` for the exhaustive model-usage map (every LLM call site, its selection, where the spend lands).

## Architecture (ports & adapters)

```
packages/
Ōö£ŌöĆŌöĆ core/         pure domain: entities, ports, use cases, prompts ŌĆö depends on `effect` + `@effect/ai`
Ōö£ŌöĆŌöĆ adapters/     Layer impls of ports ŌĆö depends on @xandreed/sdk-core + external SDKs (@effect/ai-google, @effect/ai-openai, Postgres)
ŌööŌöĆŌöĆ cli/          the efferent CLI driver: runs agents ŌĆö TUI ┬Ę print ┬Ę json ┬Ę rpc ┬Ę daemon
```

**Dependency direction is strictly inward.** `cli` ŌåÆ `adapters` ŌåÆ `core`. `core` imports nothing from siblings. `adapters` imports `@xandreed/sdk-core` + the external SDK it wraps. Drivers compose the layers at the very edge and hand off to `BunRuntime.runMain`.

## Conventions

- **Ports** are `Context.Tag` services in `@xandreed/sdk-core/ports/`. Each ships its tagged errors next to it.
- **Adapters** provide one `Layer.effect` per port. External promises go through `Effect.tryPromise`, mapped into the port's tagged error. Config via `Config.string` ŌĆö never hardcode.
- **Use cases** live in `@xandreed/sdk-core/usecases/` returning `Effect.Effect<A, E, ŌĆ”>`. No IO; the only SDK allowed in `core` is `@effect/ai` (provider-agnostic ŌĆö `LanguageModel`, `Tool`, `Toolkit`, `Prompt`). Provider packages (`@effect/ai-google`, `@effect/ai-openai`) live in `adapters`.
- **Agent configs** (`@xandreed/sdk-core/usecases/{agentConfig,coderAgentConfig}.ts`) bundle a system prompt + an `@effect/ai` `Toolkit` into an `AgentConfig<Tools>`. `runAgent` is parameterized by config; the CLI picks `coderAgentConfig(cwd)`. The toolkit's handler `Layer` (`codingToolkitLayer(cwd, skills, { allowBash })`) is provided at the driver's composition root ŌĆö it carries the runtime deps (`cwd`, `FileSystem`, `Shell`).
- **Schema** lives in `effect` itself: `import { Schema } from "effect"`.
- Bun runs `.ts` directly. No build step, no emit. `tsc --noEmit` is purely a typecheck gate.
- **No `try/catch`, `throw`, or `.catch()` in `@xandreed/sdk-core/src/`.** Error handling uses Effect's typed errors (`Effect.fail`, `Effect.die`, `Effect.catchAll`, `Effect.catchTag`). Enforced by `scripts/banTryCatch.ts` ŌĆö a zero-dep TypeScript-AST scan (so `Effect.try`/`Effect.tryPromise`/strings/comments never false-positive), folded into `bun run typecheck`.
  - **After the task is complete, run `bun run typecheck` to validate the ban.** It runs tsc + the AST scan; a banned construct fails the command and the change is rejected.
  - Enforced at three layers: a local `pre-commit` hook (installed by `bun install` via the `prepare` script ŌåÆ `scripts/installHooks.ts`; `--no-verify` bypass discouraged), the `ci` workflow on every push/PR (`.github/workflows/ci.yml`), and the release gate. Branch protection on `main` requires `ci`, so a violation can't be merged or pushed.
- File naming: camelCase for files that export functions; PascalCase for files that export types / `Context.Tag` classes.

Credentials & setup. **Credentials come *only* from `~/.efferent/auth.json`, written by the in-app `:login` flow ŌĆö there is no env-var key reading and no `init` command.** `efferent` always boots straight into the TUI; with no credential it shows `No models available. Run :login ŌĆ”` and sending a message short-circuits to the same hint. **`:login`** is an in-app flow (`packages/cli/src/cli/presentation/loginFlow.ts` + `promptBox.ts`, composited like the `:model` picker): pick *Use a subscription* (OAuth) or *Use an API key* ŌåÆ pick a provider (status-tagged Ō£ōapi key / Ō£ōsubscription / ŌĆóunconfigured) ŌåÆ paste a masked key **or** run the OAuth flow ŌåÆ persisted to `auth.json` ŌåÆ **usable that turn, no restart**. **`:logout <provider>`** removes a credential. The first successful login also pins that provider's default model. Non-interactive modes (print/json/rpc) can't run `:login`, so they require a credential already in `auth.json` (else they exit with a hint).
- **`AuthStore`** port (`@xandreed/sdk-core/ports/AuthStore.ts`) + `LocalAuthStoreLive` (`adapters/src/auth/local.ts`): per-provider creds (`{ type: "api_key" } | { type: "oauth", ŌĆ” } | { type: "local", baseUrl? }`; legacy flat-string entries read as api keys), atomic `0600` writes. `resolveKey` is read lazily per request by the router/registry, refreshing a near-expiry OAuth token first. Reads no env. **Config/auth scoping** (shared with `SettingsStore`, resolved by `adapters/src/configRoots.ts`): **`EFFERENT_HOME` set ŌćÆ that one dir is the entire world** ŌĆö `$EFFERENT_HOME/.efferent/{auth.json,config.json}`, no `~/.efferent`, no cwd, no merge (the hermetic sandbox / `start:fresh`). **Unset ŌćÆ two tiers** ŌĆö global `~/.efferent` + local `<cwd>/.efferent`, merged **local-over-global**; onboarding picks the scope (step 1), `:login` defaults global, `:set`/`/model`/`:theme` default local, and a local write seeds a `.efferent/.gitignore` (`auth.json`, `config.json`, `*.db*` ŌĆö `skills/` stays committable). The router builds the provider client per turn from `resolveKey`, so a login mid-session takes effect immediately. Evals/CI use `EnvAuthStoreLive` (`adapters/src/auth/env.ts`) ŌĆö the *only* place provider key env vars are read.
- `EFFERENT_DB_URL` ŌĆö **optional.** Selects the conversation store: `postgres://ŌĆ”` ŌåÆ Postgres; any other value ŌåÆ SQLite at that path (optionally `sqlite:`-prefixed); unset ŌåÆ zero-config **SQLite** at `~/.efferent/efferent.db` (the default; no Docker). May also live as `dbUrl` in `~/.efferent/config.json`; the env var **overrides** the config value (seeded into the env at boot by `seedDbUrlFromConfig`, since the store is chosen at layer-build before settings load). Parsed by `parseDbTarget` and selected in `adapters/src/database/migrator.ts` (`ConversationStoreLive`); migrations load via `Migrator.fromRecord` (bundle-safe).
- Optional `EFFERENT_MODEL` ŌĆö seeds the active model when no `config.json` pins one. Accepts `"<provider>:<modelId>"` (e.g. `openai:gpt-4o`) or a bare id (provider inferred; defaults to Google). Default `google:gemini-3.5-flash`. An explicit `/model` switch is persisted to `.efferent/config.json` and wins over `EFFERENT_MODEL`.

## Coding tools (CLI)

An `@effect/ai` `Toolkit` in `@xandreed/sdk-core/usecases/codingToolkit.ts`, backed by the `FileSystem` and `Shell` ports. Each tool is a `Tool.make` def with an **object** `success` Struct + a shared `failure` Struct and `failureMode: "return"` (so a tool failure is returned to the model as data, not thrown). Handlers live in `codingToolkitLayer(cwd, skills, { allowBash })`, which resolves `FileSystem`/`Shell` from context at layer-build time. (Gemini rules learned the hard way: every tool needs Ōēź1 parameter, and `success` must be an object ŌĆö see the spike notes in the plan.)

| tool         | parameters                                       | implementation                              |
|--------------|--------------------------------------------------|---------------------------------------------|
| `read_file`  | `{ path; offset?; limit? }`                      | `FileSystem.read`                            |
| `write_file` | `{ path; content }`                              | `FileSystem.write`                           |
| `edit_file`  | `{ path; edits: [{ oldText; newText }] }` *or* flat `{ path; oldText; newText }` | `normalizeEdits` ŌåÆ read ŌåÆ string replace ŌåÆ write, returns diff |
| `Bash`       | `{ command; timeout? }`                          | `Shell.exec`, cwd bound at tool-build time  |
| `grep`       | `{ pattern; dir?; flags?; context? }`            | shells out to `grep -rnE` for now            |
| `glob`       | `{ pattern; dir? }`                              | `FileSystem.glob` via `Bun.Glob`             |
| `ls`         | `{ path?; recursive? }`                          | `FileSystem.list`                            |
| `web_fetch`  | `{ url; maxBytes? }`                             | `Http.get`; HTML reduced to text             |
| `search_web` | `{ query }`                                      | `WebSearch.search` ŌåÆ `{ answer; sources }`   |
| `run_agent`  | `{ name; folder; task; seedFromNode?; seedMode?; agent?; instructions?; tools?; role? }` | spawns a folder-scoped sub-agent **in the background** and returns `{ nodeId; name; status: "running" }` immediately ŌĆö non-blocking (see Sub-agents & context tree); `name` (2-5 words) is its display title; `role` (`general`\|`code`) picks the model TIER, never a specific model |
| `wait_for_agents` | `{ nodeIds?; timeoutSeconds? }`             | the non-blocking **gather**: parks the caller (interruptibly) until a watched agent finishes / a message lands in the caller's inbox / the timeout, then returns each agent's status (+ finished summaries/files), the caller's drained inbox, and the blackboard tail. `AgentBus.awaitChange` |
| `update_plan` | `{ steps: [{ step; status }] }`                  | the agent's working plan (statuses pending/active/done); each call replaces it whole ŌĆö pure echo handler, the UI reads the args |

All paths resolved relative to the `cwd` bound in `codingToolkitLayer(cwd)`. **`edit_file` accepts two shapes:** the canonical `edits: [{ oldText, newText }]` array, or a flat single-edit `{ oldText, newText }` at the top level ŌĆö the shape models trained on Claude Code's `Edit` tool emit when there's only one edit. Both are normalised by `normalizeEdits` *inside* the handler. This matters because `@effect/ai` decodes tool-call parameters **before** the handler runs (`Toolkit.handle`), so a shape mismatch is an `AiError.MalformedOutput` that `failureMode: "return"` can't catch ŌĆö it aborts the whole turn. Keeping the parameter schema permissive enough to decode both shapes, then validating in the handler, turns "wrong shape" into a graceful, model-visible `EditFailed` instead of a dead turn. **Names matter:** Anthropic reserves the lowercase tool names `bash` / `web_search` / `computer` for its provider-defined tools (case-sensitive lookup); when we register a handler-backed tool by one of those names, the Anthropic SDK reroutes it to the built-in provider tool ŌĆö which isn't in our toolkit, so the turn fails. So our shell is **`Bash`** (capital B) and our search is **`search_web`** (reversed). `search_web` is provider-native (see below) ŌĆö it has no parameter-less constraint issue because `{ query }` satisfies the Ōēź1-parameter rule.

## Web search

Provider-native, no extra key. The `web_search` tool is backed by the `WebSearch` port (`@xandreed/sdk-core/ports/WebSearch.ts`) and `WebSearchLive` (`adapters/src/llm/webSearch.ts`). Each call is a **dedicated, grounding-only** `generateText` against a provider's *server-side* search tool ŌĆö Gemini `GoogleTool.GoogleSearch` or OpenAI `OpenAiTool.WebSearch` (`Tool.ProviderDefined`, handler-free). It returns `{ answer, sources }` (sources from the response's `UrlSourcePart`s). The model finds with `web_search`, then reads a chosen source with `web_fetch`.

**Deliberately a separate thing** (not merged into the agent's main toolkit): the search call carries *only* the search tool, never the agent's function tools ŌĆö so it needs no extra credential beyond a logged-in provider, and sidesteps providers (notably Gemini) that won't combine grounding with function calling in one request. It's also decoupled from the chat `/model`: configured via `EFFERENT_SEARCH_MODEL` (`<provider>:<modelId>`), else defaults to whichever of Google/OpenAI is logged in (Google preferred). `WebSearchLive` resolves its key per call from the `AuthStore` and builds its own client over a `FetchHttpClient`. Gemini's grounding source URLs are `vertexaisearchŌĆ”` redirects ŌĆö `web_fetch` follows them.

## Agent loop

One use case ŌĆö **`runAgent(config, conversationId, prompt, hooks?)`** in `@xandreed/sdk-core/usecases/runAgent.ts` ŌĆö drives the whole interaction. The loop lives in `@xandreed/sdk-core/usecases/agentLoop.ts`. **`@effect/ai` resolves a single model step's tool calls (its handlers are our Effects) but does NOT iterate across turns ŌĆö so iteration is ours**: each turn maps the message buffer to a `Prompt`, calls `LanguageModel.generateText({ prompt, toolkit })`, appends the response parts as the new tail, and re-invokes until `finishReason !== "tool-calls"` or `maxSteps`.

**The root is a fast, direct coder ŌĆö not a forced router.** Every mode builds the root scope from `coderPrompt` (`main.ts` ŌåÆ `rootScope`), so the root has the full coding toolkit and just does the work in its own loop ŌĆö a bug fix, a single-area feature, a question. Delegating to the background fleet is **opt-in**, governed by a prompt-only `# When to delegate` policy (`renderDelegationPolicy` in `packages/cli/src/prompts/coder.ts`, emitted only when a `coordinator`/`research-coordinator` lead is in the roster): spin up a fleet only on **breadth** (Ōēź3 independent areas), **scale** (a change spanning many packages that would blow the context window serially), or **async intent** (fire-and-forget / a scheduled, unattended job) ŌĆö otherwise do it yourself. (The old design silently ran every mode through a thin "triage and dispatch" router root that forbade the agent from doing work; `rootPrompt`/`rootSystemPrompt`/`renderDelegationSection` are deleted.)

```
efferent <prompt>
  Ōåō
runAgent(coderAgentConfig(cwd), conversationId, prompt, hooks)
  Ōö£ŌöĆŌöĆ ConversationStore.append(user) ; list ŌåÆ message history
  Ōö£ŌöĆŌöĆ runAgentLoop({ system, messages, toolkit, hooks, maxSteps })
  Ōöé     Ōö£ŌöĆŌöĆ turn 0..N: onTransformContext? ŌåÆ onTurnStart
  Ōöé     Ōöé              ŌåÆ Prompt.make([system, ...toPromptMessages(messages)])
  Ōöé     Ōöé              ŌåÆ LanguageModel.generateText({ prompt, toolkit })   (resolves this step's tools)
  Ōöé     Ōöé              ŌåÆ responseToAgentMessages(res.content) onto buffer  (carries thought_signature in providerOptions)
  Ōöé     Ōöé              ŌåÆ re-emit onBeforeToolCall / onAfterToolCall / onAssistantMessage(TokenUsage) from the response
  Ōöé     Ōöé              ŌåÆ continue iff finishReason === "tool-calls" && toolCalls > 0 ; onShouldStopAfterTurn?
  Ōöé     ŌööŌöĆŌöĆ while turnIndex < maxSteps
  Ōö£ŌöĆŌöĆ ConversationStore.append(tail)
  ŌööŌöĆŌöĆ return AgentResult { finalText, messages }
```

**Prompt mapping** (`@xandreed/sdk-core/usecases/promptMapping.ts`): bridges our persisted `AgentMessage` (our stable shape, unchanged) with `@effect/ai`'s `Prompt`/`Response`. The opaque provider blob is carried verbatim both ways (`providerOptions Ōåö options`/`metadata`), which is how Gemini's `thought_signature` round-trips across our turns ŌĆö `Prompt.fromResponseParts` drops it, so we map by hand.

**Hooks** (`@xandreed/sdk-core/AgentHooks`): the loop re-emits the legacy event vocabulary (`onTurnStart` / tool events / `onAssistantMessage`) from each resolved response, so the CLI's `makeEventHooks(queue)` (`packages/cli/src/events.ts`) and the TUI execution tree keep working unchanged.

**Graceful tool errors**: each tool's `failureMode: "return"` + `failure` Struct means a handler failure (e.g. `FileNotFound`, ambiguous edit) is returned to the model as a tool result instead of aborting the turn. **But `failureMode` only catches *handler* failures** ŌĆö `@effect/ai` decodes a tool call's parameters (and resolves the tool name) *inside* `Toolkit.handle`, **before** the handler runs, so a wrong-shaped call or a hallucinated tool name fails with `AiError.MalformedOutput`, which `failureMode` never sees and which aborts the whole turn. `recoverMalformedToolCalls` (`agentLoop.ts`) closes that gap: it resolves the toolkit's handler once and wraps it so a `MalformedOutput` is converted into an ordinary tool *result* (`isFailure: true`, `error: "InvalidToolCall"`). The assistant tool-call Ōåö tool-result pairing stays valid, the loop proceeds to its next turn, and the model reads the decode error from context and emits a corrected call ŌĆö the **same** recovery path as any returned failure, no retry machinery. (`MalformedInput` ŌĆö a result encode/validate failure, i.e. *our* bug ŌĆö is deliberately let through, not masked.)

**Bash safety**: three layers. The static `allowBash` flag (denied ŌåÆ returned as a tool failure; non-interactive modes pass `--allow-bash`), then the **`Approval` port** (`@xandreed/sdk-core/ports/Approval.ts`) consulted per command, fronted by the **fast-tier auto-approval judge**. An unmatched command first goes to the judge (`judgeApproval` in `core/usecases/autoApproval.ts`, `UtilityLlm` on the **fast** role; `:set autoApprove off` disables, default ON): it classifies the input against the **permitted folders** ŌĆö the workspace root (opening efferent in a cwd is the standing grant) plus every folder granted before ŌĆö and either **allows silently** (ordinary development work whose paths stay inside; a dim `fast approved: ŌĆ”` info line in the rail, spend ŌåÆ `byRole.fast`) or **prompts**, the modal carrying its one-line reason. Not a security boundary (the scope sandbox owns that) ŌĆö a prompt-fatigue killer: the judge can only remove prompts, never a human's view of anything it isn't sure about, and any error (no key, bad JSON, 429) degrades to the modal. The TUI modal (`cli/approval.ts`, `a/s/p/d`): **allow once / allow for session / always allow in this project / deny with a typed reason**. Permission is **path-based**: when the judge named the out-of-bounds folder, session/project answers grant *that folder* (session Ref / `Settings.approvedFolders`) and future commands inside it pass the judge; with no folder hint they grant the command *rule* as before (`bashRuleKey`: command+subcommand, exact-match when shell metacharacters are present ŌĆö persisted to `Settings.approvedBashRules`). A denial (with its reason) returns to the model as an ordinary tool failure, so it adjusts in the same turn. Evals/CI provide the static allow-all impl behind the `--allow-bash` gate ŌĆö CI never prompts and never judges. **Unattended (scheduled) runs are different** ŌĆö they can't prompt a human, but blindly allowing everything is unsafe, so the cron path uses a **headless parking approval** (`makeHeadlessApproval`, `packages/cli/src/workspace/headlessApproval.ts`): the fast-tier judge still auto-approves ordinary in-scope work, but anything it can't clear emits a **`needs_human`** event (`parked: true`) and is **denied-with-reason** (returned to the model, never silently allowed). Interactive approval prompts emit the same event with `parked: false`, so a top-level "decisions need you" roster can surface both (see CLI shape ŌåÆ `DecisionsBar`). This replaces the old blind `ApprovalAllowAllLive` on the cron path.

## Sub-agents & context tree

Sub-agent spawning is **one generic tool over a persistent branching context tree**, and it is **asynchronous** ŌĆö the fleet runs like a team of people, not a blocking call tree. **The whole orchestration substrate now lives in `@xandreed/sdk-core`** ŌĆö `buildScopeRuntime.ts`, the unified `agentBus.ts`, `codingToolkit.ts`, `discoverScopeTree.ts`, `loadTools.ts`, `schedule.ts`, `staleness.ts`, and `parseFrontmatter.ts` are all `core/usecases/`, and the shared prompt pieces (`sections.ts` = `subAgentsSection`/`coordinationSection`/`renderAgentsSection`/`renderMemorySection`; `scopeAgent.ts` = `renderScopeSystemPrompt`) are `core/prompts/`. The loop primitives they call (`runAgentLoop`, `RunContextRef`, `tokenBudget`) are core too. The bus is the **Supervisor** ŌĆö the orchestration substrate, owned by core (there's an exported `type Supervisor = AgentBus`). It is **threaded as a per-session value, NOT a `Context.Tag` port**, because it's per-session stateful and carries a per-runtime event sink (`onBusEvent`) ŌĆö a single shared Layer would fight that.

- **`run_agent({ name, folder, task, seedFromNode?, seedMode?, agent?, instructions?, tools?, role? })`**: spawns a sub-agent scoped to `folder` (writes + bash confined via `ScopeBinding{rootDir, enforceWrite}` ŌĆö the sandbox machinery in `codingToolkit.ts`, unchanged) and **forks it as a supervised background fiber** (`Effect.forkDaemon`, registered on the bus), returning `{ nodeId, name, status: "running" }` **immediately** ŌĆö the caller never blocks on the subtree (that was the "parent hangs while the fleet works" bug). `failureMode: "return"` so a failed spawn is data, not a dead turn. The toolkit is **static** (`genericToolkit` = base tools + comms + `run_agent` + `wait_for_agents`) ŌĆö the same toolkit at every depth; sub-agents can spawn sub-agents.
- **Gather, don't block** (`wait_for_agents`, `AgentBus.awaitChange`): a spawner collects results with `wait_for_agents({ nodeIds?, timeoutSeconds? })`, which parks **only the caller's fiber**, interruptibly, until a watched agent finishes, a message lands in the caller's inbox, or the timeout ŌĆö then returns each agent's status (+ finished summaries/files), the caller's drained inbox, and the blackboard tail. No agent ever blocks another; loop it until `allDone`. The timeout default is **10s** (was 60s; clamped 1ŌĆō300) ŌĆö the bus wakes the waiter instantly on real progress (a finish / an inbox message), so the timeout is just the idle ceiling and a tighter one keeps a stuck waiter responsive. When a run finishes, the bus also drops a completion line in its **parent's mailbox** + the blackboard and keeps a terminal result record, so a parent that wasn't waiting still picks it up at its next turn boundary.
- **The bus is the live layer** (`agentBus.ts`, `AgentBus`): per-agent **mailboxes** (keyed by node id; the root conversation registers one under its `conversationId`, so the human can message the lead mid-turn), a shared **blackboard**, plus **supervision** ŌĆö each running agent's fiber (so `:stop`/Esc/teardown interrupt it via `interruptAll`), a completion `Deferred` (so a parent awaits without polling), parentŌåÆchild links (`childrenOf`, running + finished), and a `snapshot`. The durable record stays the context tree; this is the live layer the TUI + agents tap (`:fleet`, `:tree`, the header `ŌŚå N agents` chip, a node preview).
- **The built-in team** (`teamAgents.ts` in `packages/cli/src/usecases/`, merged by `withBuiltinAgents`): when the root *does* decide to delegate (the opt-in triggers above), it hands the work to a **`coordinator`** (leads ŌĆö `run_agent` + `wait_for_agents` + comms, no writes), which staffs the specialists the work needs ŌĆö **`frontend` / `backend` / `qa` / `product` / `implementer`** (leaf coders + comms) ŌĆö and validates each piece with the read-only **`architect`** (fresh context, SOUND/NEEDS WORK/BLOCKED) before delivering. The team is unchanged from before ŌĆö only the *trigger* changed (delegation is opt-in, not mandatory). A workspace `.efferent/agents/<name>.md` overrides any built-in.
- **Persistence** (`ContextTreeStore` port, `@xandreed/sdk-core/ports/ContextTreeStore.ts`; SQLite migration `0002` / pg `0006`, tables `context_nodes` + `context_messages`): every spawn is an `AgentContextNode` `{ parentId, rootConversationId, edgeKind, folder, seed, status, returnSummary, filesChanged, usage }`. The node's full message history persists; the `seed` column is a small **descriptor** (`task` | `selection` | `handoff`) ŌĆö actual seed messages live only in `context_messages`.
- **Resume / branch / handoff**: `seedFromNode` + `seedMode: "resume"` appends the task to the existing node and continues in its context (same node grows ŌĆö full history every turn); `"branch"` spawns a **child** node (`edgeKind: "branched"`) seeded from the source node's verbatim messages; `"handoff"` (prompted as the PREFERRED follow-up mode) spawns a child seeded with a **generated brief** of the source's work (`generateHandoffBrief` ŌåÆ `handoffToMessage`, seed kind `handoff`) ŌĆö continuity at a fraction of the tokens. The routing policy lives in the tool description + `# Sub-agents` prompt section: one agent = one piece of work, fresh spawn by default, reuse only for direct follow-ups, cheapest sufficient seed. **Context-passing**: the spawner is the sub-agent's ONLY bridge (it sees only its `task` + folder + SCOPE.md + project knowledge ŌĆö not the conversation, the user's goal, the plan, or siblings' findings), so the prompts demand a full brief in `task` (OBJECTIVE / CONTEXT / OUTPUT / BOUNDARIES, Anthropic's multi-agent protocol). As a structural backstop, every fresh/handoff spawn is seeded with a one-line **mission** preamble from `RunContext.mission` (the human's request, inherited down the subtree) so a leaf never works fully blind, and spawned workers are nudged (`core/prompts/sections.ts`) to **read the blackboard FIRST** ŌĆö a sibling may have already posted a finding/contract that changes the work. **Model**: a spawn picks a model TIER via `role` (`general` default, `code` for writing code) or the agent definition's `role:` frontmatter ŌĆö never a specific model (the old per-agent `model` footgun is gone; see Models & providers). The tree records how context branched and evolved.
- **Identity & bounds**: `RunContextRef` (a `FiberRef` in `usecases/runContext.ts`) carries `{ rootConversationId, parentNodeId, depth, tokenPool }` ambiently ŌĆö `runAgent` seeds it via `Effect.locally`, each spawn re-seeds for its children. A `maxDepth` guard (default 2 ŌĆö the root agent can spawn sub-agents, and those can spawn one more level) returns `MaxDepthReached` as a tool failure; each sub-agent run is bounded by its own step cap (default 80, `Settings.subAgentMaxSteps`, `:set subAgentMaxSteps <n>`, threaded live through `RunContext` like the pool). A capped run's summary is stamped `[stopped early: the step limit was reached ŌĆö this result is partial]` (mirroring the budget stop), so the parent reads truncation instead of trusting a mid-thought sentence as the deliverable.
- **Token budget** (`usecases/tokenBudget.ts`): depth bounds termination, the **shared pool** bounds spend ŌĆö one `Ref` (default 1M billed tokens, `Settings.subAgentTokenBudget`, `:set subAgentTokenBudget <n>`, 0 = off) drawn from by every sub-agent in a turn's subtree. A drained pool refuses new spawns (`BudgetExhausted`, model-readable) and stops running sub-agents at their next turn boundary, marking the summary `[stopped early ŌĆ”]` so partial results read as partial. `:tree` rows show each node's billed tokens.
- **Parallel fan-out & write-sequencing**: spawning is non-blocking, so multiple `run_agent` calls in one assistant turn each fork-and-return at once ŌĆö the whole fleet runs concurrently (a turn's tool calls themselves resolve with `generateText`'s `concurrency`, default 4 ŌĆö `DEFAULT_TOOL_CONCURRENCY` in `agentLoop.ts`). **There is no folder lock.** Read-only work (research / investigation / review) fans out in parallel ŌĆö readers never conflict; **writing is sequenced by the ORCHESTRATOR**, not a lock: the coordinator protocol is *investigate in parallel ŌåÆ implement one writer at a time* (spawn a coder, `wait_for_agents`, spawn the next). The old per-folder semaphore held for an agent's whole run, so a hung writer stranded its same-folder siblings invisibly + unrecoverably (it's what wedged a live fleet of three read-only reviewers); sequencing at the orchestrator keeps control where it's observable (`wait_for_agents` timeouts) and recoverable. Because the fleet are background daemons (they outlive the spawning turn), teardown is explicit: **Esc** interrupts the root fiber AND `bus.interruptAll()`, and a runtime finalizer does the same on exit ŌĆö structured cancellation, no orphans. **Wedged/dead nodes are handled too**: the spawn body wraps in `Effect.onExit` (`buildScopeRuntime.ts`) so a run that exits *without* recording a terminal return (interrupted / died) records an `error` return and notifies the parent immediately ŌĆö the DB node never strands as `running`. A complementary **mid-session sweeper** (`inProcess.ts`, ~120s grace via `SWEEP_GRACE_MS`, `isNodeStranded`) flips any still-`running` node that's no longer live on the bus to `error` and notifies the parent, closing the gap the startup-only reconcile left. The Activity pane attributes parallel runs correctly: inner tool/assistant events carry `subAgentNodeId` (stamped by `makeInnerHooks`), `subagent_start` carries `parentNodeId`, and the pump keys each run to its own tree container (`onSubAgentStartKeyed`/`onToolStartUnder` in `executionTree.ts`) instead of the old single-open-stack model that nested parallel runs into a staircase. The events flow through the pump even for background daemons, so `:tree`/activity/the fleet chip update live after the root turn ends.
- **Staleness** (`usecases/staleness.ts`): a persisted context is a cache of a world that keeps changing. Nodes are stamped with the workspace git HEAD at finish (`workspaceRef`); resume/branch against a moved HEAD prepends a **staleness brief** (ref range + `git diff --stat` scoped to the node's folder + "re-read before editing") and `:tree` shows a yellow `stale` badge. All best-effort: non-git workspaces never stamp, a GC'd ref never breaks a spawn.
- **`SCOPE.md` is ambient folder context, not an agent definition**: `getScopePromptBody(folder)` (`discoverScopeTree.ts`) reads `folder/SCOPE.md` (frontmatter stripped) and injects the body into any sub-agent scoped there. Frontmatter `name`/`description` are inert metadata now; the old `delegate_to_<name>` tools are gone.
- **Events**: `onSubAgentStart/End` carry an optional `nodeId`, bridging the ephemeral per-run execution tree and the persistent context tree. In the TUI rail, a `run_agent` call renders as the sub-agent container itself (Task-style), not a redundant tool pill.
- **TUI** ŌĆö the right pane is ONE always-visible **fleet tree** (`:tree`, `FleetTree` ŌåÆ `ContextTreeView` + a reactive **detail section** below, `view/panes/side/NodeDetail.tsx`, following the cursor ŌĆö full unclipped summary/seed/files/billed tokens, a running node's **live tool feed** via `TreeNode.nodeId` + `findByNodeId`). It is **current-session-only** and **always expanded**: a single **non-foldable root** ŌĆö the active session (its orchestrator shows a live `ŌŚÅ` while its own turn runs) ŌĆö with its coordinator/agents/sub-agents nested beneath as a **git-graph-railed tree** (`Ōö£ŌöĆ`/`ŌööŌöĆ` + `Ōöé` continuation, computed purely in `buildNavRows`; a `branched` connector is accent-tinted), each node a status glyph (`ŌŚÅ`/`Ō£ō`/`Ō£Ś`), provenance (`spawned`/`branched`/`resumed`), seed kind, files count, return summary, billed tokens. Loaders (`loadAgentTree` / `loadSessions` / `refreshNav` in `actions/contextTree.ts`, **`activeOnly` by default**) seed it at boot and refresh each turn end + on every sub-agent start/end ŌĆö so the fleet appears and flips status live (the in-process `code` driver AND the remote `efferent` daemon path, which reads the shared store via `refreshNav` in `remoteRuntime.ts`). Other sessions are reached via the **`:browse`/resume** picker, not this pane ŌĆö **no four-view cycle, no `sessions`/`fleets` tabs** (that model is gone). **`ŌåĄ`** opens the row: a conversation **becomes the active session** (`switchToConversation` ŌĆö resume + reload, refused mid-turn); an agent node opens a read-only **session preview** overlaying the conversation pane (`NodePreview` in `state/conversation.ts` ŌĆö `viewBlocks = preview ?? blocks`, so the event pump keeps writing to the live rail underneath; title `agent: <folder>`, `ŌöĆŌöĆ seed ŌĆ” ŌöĆŌöĆ`/`ŌöĆŌöĆ run starts ŌöĆŌöĆ` boundary rules from `seedMessageCount`; `ŌåĄ`-again/`q`/idle-`Esc` close). **While a preview is open the composer routes to that agent ŌĆö like pairing with a teammate** (input title `input ŌåÆ agent: <folder>`), and it never freezes: if the node is **running**, the message is delivered to its **bus mailbox** (`bus.post` ŌĆö it reads it at its next step, the composer stays free, no busy flip); if it's **finished**, the message **resumes the node in place** via `ScopeRuntime.resumeNode` (the human-driven `seedMode:"resume"` ŌĆö staleness brief, depth-0 fresh budget, children hang off the node; refused while another turn runs). The preview re-fetches at turn end (`{focus:false}` refresh keeps the composer). **Routing is explicit: with a preview open the composer pairs with that agent (above); with no preview open, a message typed while the root is busy QUEUES (agy-style ŌĆö a pending `Ō¢Ė` entry that runs as the next turn when the current finishes), NOT injected mid-turn.** The in-process `code` driver queues locally (`submit.ts` ŌåÆ `run.enqueue`, drained in `finishTurn`); the remote `efferent` driver queues on the daemon (`inProcess.send` ŌåÆ run-state queue, surfaced in `SessionState.queue`, mirrored into the pending list by `remoteRuntime`'s `refreshQueue`). ## Conversation rail = a keyed cache (single source of truth)

The TUI rail (`state/conversation.ts` `blocks`) is a **normalized, identity-keyed cache** ŌĆö the React-Query model: every writer (the live event pump, the optimistic submit, a DB re-projection on resume/attach/resync) addresses a block by **stable identity**, so the same logical message can't appear twice no matter how many writers touch it (the old duplicate-messages bug). Identity is the message's **absolute store position**: `append` returns it; `messageKey(position, "u"|"a"|"r", ord)` (`presentation/conversation.ts`) is computed **identically** by the live pump (off the event's `position`) and by `projectHistory` (off `baseOffset + index`, `baseOffset` = the window's absolute start, so it's **handoff-safe** ŌĆö an absolute position never shifts when `listActive` narrows). `pushBlock` upserts by identity (a replay/re-projection lands on the same entry); `reconcile` (the reconnect-resync write) merges by key and keeps any live-but-unpersisted tail (never drops the streaming reply); `setBlocks` stays a blind replace for true document swaps (`:clear`/build). The user line flows through a **`user_message` event** the daemon emits from `runAgent` (carrying `position`); an optimistic line (`pushOptimisticUser` ŌåÆ `opt:<n>` + a FIFO of pending keys) reconciles onto its authoritative position by `resolveOptimisticUser` ŌĆö never by content-hash (two identical-text messages stay distinct). The remote reconnect-resync (`remoteRuntime.ts`) is **idle-gated** (tail-only while a turn streams) with a strictly-exclusive cursor, so a blip can't drop or double the rail. Contract proof: `state/conversation.test.ts` (idempotent upsert, optimistic reconcile, reconcile-preserves-tail, cross-writer key equality, handoff-safety).

**Daemon-authoritative phase**: `SessionState` now carries a coarse lifecycle `phase` (`idle`/`thinking`/`tool`) derived by a **pure core reducer** (`reducePhase` in `core/usecases/agentPhase.ts`) the daemon folds every published *root* event through (sub-agent inner events + `run_agent` are skipped ŌĆö they're the orthogonal fleet). The client reconciles its richer `agentState` machine against `getState(id).phase` on (re)attach + resync ŌĆö so a client joining mid-turn shows `thinking` and an idle daemon clears a stale spinner, fixing the phantom-"thinking"-on-reattach bug. The CLI's `agentState.ts` reuses the same `AgentPhase` literal + transition rules verbatim, so the two never drift. **Tool-pill identity is deterministic**: `ensureToolCallIds` (`promptMapping.ts`) mints a stable id (`<turnIndex>:<toolName>:<ordinal>`) at the loop source for providers (notably Gemini) that omit tool-call ids, writing it into the response BEFORE it splits into events + the persisted `AgentMessage` ŌĆö so live key === projected key and re-projection/re-attach never duplicates or reorders a pill.

## Handoff & conversation context

A **handoff** replaces the *loaded* history with a model-generated summary while keeping the originals. It's how a long session frees context without losing the record.

- **Storage** (`checkpoints` table, migration `0005`): a checkpoint is `{ messagePosition, summary }` for a conversation. Original `messages` rows are never modified. `ConversationStore` exposes `checkpoint(id, summary)` (atomic ŌĆö folds at `COALESCE(MAX(position), -1)` in one statement), `getLatestCheckpoint`, `listCheckpoints`, `list` (ALL messages, for browsing), and `listActive` (only real rows **after** the latest fold).
- **Loading** (`runAgent`): loads `getLatestCheckpoint` + `listActive`, prepending the summary as a single synthetic `user` message via `handoffToMessage` (`promptMapping.ts`) ŌĆö domain logic stays in core, not the adapter. So the model sees `[handoff summary] + [messages since the fold] + [new user msg]`; everything before the fold is preserved but never re-fed.
- **Creating** (`createHandoff`, `@xandreed/sdk-core/usecases/handoff.ts`): summarizes the **currently loaded view** (prior summary + active window, never the raw `list`) with `HANDOFF_PROMPT` (`prompts/handoff.ts`, a Goal/State/Next-steps/Constraints brief) and writes a checkpoint. Summarizing the loaded view keeps handoffs **cumulative** ŌĆö a second handoff folds the first summary back in. No-op when nothing new since the last fold.
- **TUI**: `:handoff` runs it (pushes a checkpoint block); `:context` toggles the **context viewer** in the side pane (`cli/presentation/contextView.ts` ŌĆö `buildContextView` partitions `list` + `listCheckpoints` into *archived* segments (folded, not loaded) and the *loaded* segment; `buildContextRows` groups each segment's messages into **foldable, selectable turns**, rendered as a tree so the replacement is visible); `:browse` lists workspace conversations and `:resume <#|id>` switches to one (full replay for browsing, active-window for execution). On boot with no `--resume`, if the workspace has prior conversations, a **startup picker** (`openConversationPicker` ŌåÆ a select overlay, purpose `conversation`) floats "Resume a conversation" over the fresh session ŌĆö a leading "’╝ŗ Start a new conversation" row then `<date> ┬Ę <title|first-prompt>` per conversation (the title is generated on the utility model after a session's first exchange ŌĆö `generateSessionTitle` ŌåÆ `ConversationStore.setTitle`); Enter resumes via `resumeConversation`, Esc / "start new" leaves the fresh session. No prior conversations ŌåÆ straight to an empty rail.
- **Curate context ŌåÆ new session**: in the context viewer, `Space` selects/deselects a **turn** *or* a **handoff** (the `ŌÜæ` archived-segment header) and `b` (or `:build`) **builds a new conversation** seeded with only the picked units, switching to it (`messagesForSelectedTurns` ŌåÆ `ConversationStore.create`/`append` ŌåÆ the `:resume` switch). A selected handoff contributes **only its summary, as one `handoffToMessage` message** (not the folded originals); a handoff and its own inner turns are therefore **mutually exclusive** (selecting one clears the other ŌĆö `archivedTurnRanges`/`handoffOwningTurn` in `contextView.ts`). Turn granularity keeps tool-call/result pairs valid; the original conversation is untouched.

## Context compaction (compression without breaking cache)

Cache-safe context compression, implemented natively in `core/usecases/compaction.ts`. The governing constraint: provider prompt caches key on a **byte-stable prefix**, so compression must never rewrite history. Four tactics:

1. **Append-time tool-result compression** (`compressToolResults`, called by `runAgentLoop` on each step's new tail): any string inside a tool-result output over the budget (`Settings.toolResultMaxTokens`, default 4000 tokens Ōēł 16k chars, 0 = off; threaded to sub-agent loops via `RunContext` like the step cap) is compressed the moment it enters the buffer ŌĆö the persisted record and every future prompt carry the compressed form from byte one, and nothing already sent is ever rewritten. Compression is **structure-aware first** (`compactionContent.ts`, the SearchCompressor/LogCompressor ideas from compaction ported to our shapes): grep-shaped output (`path:NN:` lines, any tool) is grouped per file ŌĆö every file visible, first 5 matches each, exact counts in the marker; Bash output with error signal keeps head/tail + every error with its trace block intact (a state machine absorbs `at `/`File "`/indented continuations) + deduped warnings (`(├ŚN)`) + test-runner summary lines, re-emitted in original order with inline `[ŌĆ”N lines omittedŌĆ”]` gap markers. Shapeless text falls back to the blind head (75%) + tail (12.5%) clip. **The TUI still shows the full output** (hooks emit the raw response before the tail is compressed).
2. **Reversible markers** ŌĆö the clip marker names what was dropped (`[ŌĆ”compaction: ~4509 tokens of this Bash output omittedŌĆ”]`) and how to get it back (re-read with offset/limit, narrower grep, `| head`) ŌĆö compression the model can undo on demand. Live-verified: the model greps its way to the buried line instead of trusting the hole.
3. **FAST-tier middle digests** ŌĆö when `UtilityLlm` is in context and the dropped middle is big enough, the fast role writes a Ōēż120-word summary into the marker (the fast tier's first consumer). Usage is reported via the new `onHelperUsage` hook ŌåÆ `helper_usage` event ŌåÆ `byRole.fast`. Best-effort: no service / provider error degrades to the plain marker; sub-agent loops forward the spend to the parent ledger.
4. **Threshold auto-fold** (`autoHandoffPct`, default 85, 0 = off): when a finished turn's context crossed the threshold, the TUI runs `:handoff` automatically at the boundary ŌĆö one deliberate prefix rebuild (the only cache-safe way to shrink history), then the cache is warm again. Skipped while messages are queued and on `~` resume estimates.

Note: `read_file`/`web_fetch` carry their own per-tool caps (`truncated: true`); compaction is the *generic backstop* for anything without one ŌĆö bash stdout, grep floods, future tools. Verified live: an 80k-char `cat` persisted at 14.6k chars with marker + digest, `╬Ż` showing all three tiers.

## Models & providers (multi-provider router)

The agent loop talks to one provider-agnostic `LanguageModel`; which provider/model backs it is a **runtime selection**, not a compile-time layer choice.

- **Router** (`adapters/src/llm/router.ts`, `RouterLanguageModelLive`): a `LanguageModel` whose `generateText`/`streamText`/`generateObject` read `ModelRegistry.current` on every call, resolve a key from the `AuthStore` (refreshing an expired OAuth token first), and build the chosen provider's `@effect/ai` service **per request** from that key over a shared `FetchHttpClient` (`Effect.scoped`/`unwrapScoped` so the client lives exactly one call). So a credential added mid-session via `:login`, or a `/model` switch, takes effect on the next turn with no rebuild. There is no longer a `clients.ts` ŌĆö keys aren't captured at layer-build.
- **Transient-failure resilience** (`retryableLlm` in `adapters/src/llm/retry.ts`): the default provider (Kimi via the opencode gateway) returns 429 "engine overloaded" under load, and with no retry that was instant turn death and a thrashing fleet. `retryableLlm` wraps each LLM *connect* (never a live stream) and retries only **transient** failures ŌĆö 429, retryable 5xx (500/502/503/504), transport (`HttpRequestError`), and fetch timeout/abort/reset signatures sniffed from a wrapped `UnknownError`; never a 4xx client error or a decode bug (`MalformedInput`/`MalformedOutput`), which are permanent. Up to 3 retries, exponential **1s ŌåÆ 2s ŌåÆ 4s** (capped 8s) with ┬▒25% jitter, honoring `Retry-After` when present; each retry annotates the active span (`llm.retry.*`) + logs a warning. Wired at the **router** (`generateText`/`generateObject`, covering every provider incl. the Kimi/opencode gateway) plus `utilityLlm` + `webSearch`.
- **Anthropic OAuth** (subscription): when the credential is `oauth`, the client is built with `apiKey: undefined` + a `transformClient` that sends `Authorization: Bearer ŌĆ”` and `anthropic-beta: claude-code-20250219,oauth-2025-04-20` (no `x-api-key`), and the router prepends the required `"You are Claude Code, Anthropic's official CLI for Claude."` system block (`Prompt.merge`). Protocol + constants in `adapters/src/auth/oauth/anthropic.ts` (PKCE, authorize URL, exchange, refresh); callback server + browser-open in `cli/src/login/oauthServer.ts`.
- **Selection** lives in `SettingsStore` as `settings.model = "<provider>:<modelId>"` (the single source of truth, persisted to `.efferent/config.json`). `parseModel`/`formatModel`/`contextWindowFor`/`defaultModelForProvider(s)` are pure helpers in `@xandreed/sdk-core/entities/Model.ts`.
- **`ModelRegistry`** port (`@xandreed/sdk-core/ports/ModelRegistry.ts`): `current` (parsed selection), `list` (live catalogue), `select` (persist + return). `ModelRegistryLive` (`adapters/src/llm/modelRegistry.ts`) fetches the catalogue over **raw HTTP** (Google `ŌĆ”/v1beta/models`, OpenAI `ŌĆ”/v1/models`, Anthropic `ŌĆ”/v1/models`) and parses defensively; keys come from the `AuthStore`, so only logged-in providers are queried (Anthropic OAuth uses a Bearer header).
- **`ModelLive`** bundles router + `ModelRegistry` + dynamic `LlmInfo` over an internal `FetchHttpClient`; requires `SettingsStore` + `AuthStore`. Replaces the old single-provider `GoogleLive`.

**Model roles (general / code / fast)** ŌĆö one mental model, three knobs (`ModelRole` + `AgentModelRole` + `modelForRole`/`roleIsConfigured`/`selectionFromString` in `entities/Model.ts`). A running agent NEVER picks its own model ŌĆö its role is structural, and the human owns which model backs each tier (the Aider `--model`/`--editor-model`/`--weak-model` split). The three roles:

- **general** ŌĆö the root conversation / orchestrator AND research / analysis / planning sub-agents: the general-purpose brain (`settings.model`, the `/model` switch).
- **code** ŌĆö the coding sub-agents (the fleet writing code) (`settings.codeModel`; unset ŌåÆ general). A sub-agent that writes code runs here; one that researches runs on general. The **spawner picks the tier** (`run_agent({ role: "general" | "code" })` or an agent definition's `role:`), never an arbitrary model ŌĆö default `general`.
- **fast** ŌĆö one-shot helper calls off the loop (`settings.fastModel`; unset ŌåÆ general): compaction middle digests (see Context compaction), auto-approval judgments (see Agent loop ŌåÆ Bash safety), and **session titles**. Helper-only ŌĆö never a sub-agent's role. Reached through `UtilityLlm.complete(prompt, { role: "fast" })`.

The run **freezes all three role models at start** into `RunContext.pinnedModels`; the router resolves a fiber's call to `pinnedModels[role]` (the fiber's `RunContext.modelRole`, set to its spawned role), so a mid-run `/model` / `:set codeModel` can't move a running fleet ŌĆö which also keeps each provider's prompt-cache prefix warm. `UtilityLlmLive` returns `{ text, usage }` so each tier's spend is countable; per-role billed tokens accumulate in `SessionStats.byRole` (root ŌåÆ general; a sub-agent ŌåÆ its role, carried on the `assistant_message` event's `subAgentRole`; helper usage ŌåÆ fast).

**`/model`** (TUI): no arg opens the live-catalogue picker for **general** (persists, updates the status bar); **`:model code`** / **`:model fast`** open the same picker for that role, with a leading *default (follow general)* row to clear it. Direct path: `:set codeModel`/`:set fastModel <provider>:<modelId>|default`. Switching provider mid-conversation applies going forward and surfaces a one-line hint (Gemini 400s on prior non-Gemini tool-calls that lack a `thought_signature` ŌĆö `/reset` if it errors).

**Caching** is aggressive but provider-native, and the governing rule is one line: **a byte-stable prefix per agent**. Each agent keeps its OWN cache ŌĆö there's no cross-sub-agent carryover, so a sub-agent's distinct system prompt + `code`-tier model is simply its own prefix, and the `pinnedModels` freeze (above) means neither a `/model` switch nor a `:set` moves a *running* agent's model and invalidates its prefix mid-flight. The seed is stable too: the mission preamble + task are written once at spawn, never rewritten. Per provider: OpenAI gets automatic prompt-prefix caching + a stable `prompt_cache_key`; Gemini relies on implicit context caching (stable prefix ŌåÆ `cachedContentTokenCount`); **OpenCode/Kimi** (the default provider) auto-caches a stable prefix server-side, and `openCode.ts` reads the hit back (`cached_tokens` / `prompt_cache_hit_tokens`, six vendor fallbacks) so the status bar `cache%` lights up with no extra config; **Anthropic requires explicit opt-in** ŌĆö the router stamps `cache_control: ephemeral` breakpoints (`withAnthropicCacheBreakpoints` in `providers.ts`) on the last system message + the last two non-system messages every call (markers move with the tail; placement isn't part of Anthropic's content hash, so old prefixes keep hitting). Helper tiers (UtilityLlm, web search) deliberately don't stamp ŌĆö one-shot prompts would pay the +25% write premium for nothing. Anthropic's `input_tokens` EXCLUDES cache reads/writes, so `extractUsage` folds the raw anthropic metadata back in (else the gauge reads ~0 on a cached turn). Live-verified: turn 2 on Sonnet = `93% cached`. Explicit Gemini `cachedContent` is not expressible through `@effect/ai-google` today (it always sends full `contents`), so we don't fake it.

## Skills

`.efferent/skills/*.md` files are auto-discovered at startup. The search path walks `cwd ŌåÆ parents ŌåÆ ~/.efferent/skills/`; closer-to-cwd shadows farther on name collisions. Each skill file has YAML-ish frontmatter and a free-form markdown body:

```
---
name: <slug>
description: <one-line summary for the prompt>
---

(detailed procedure for the agent to follow)
```

At startup, names + descriptions are injected into the coder system prompt under a `# Skills` section. The bodies are lazy-loaded by the model via `read_skill({ name })` only when relevant ŌĆö reusable procedures you can ship without changing the code.

Loader: `loadSkills(cwd, homeDir)` in `@xandreed/sdk-core/usecases/loadSkills.ts`. Failures (missing dirs, malformed frontmatter) are silently skipped ŌĆö a broken skill never breaks the agent.

## Evals

A fourth package, **`packages/evals`** ŌĆö a minimal, Effect-native eval library: the `data ŌåÆ task ŌåÆ scorers` shape expressed as Effects, so a `task` can be the real agent loop and a `Scorer` can itself call an LLM. No sqlite, no UI, no persistence; runs under Bun directly.

- **Framework** (`src/framework/`): `EvalSpec<I,O,T,R>` is pure data (`defineEval`); `runEval(spec)` returns `Effect<EvalReport, never, R>` ŌĆö every per-case/per-scorer failure is captured via `Effect.exit` (so a provider 429 scores 0, never crashes the run) and `R` is left open for the caller to provide once. Scorers (`scorers.ts`): `predicate`, `includesAll` (substring-coverage ratio), `fromEffect`, and `llmJudge` (LLM-as-judge, parses `{"score","reason"}`). `report.ts` prints a coloured per-suite table.
- **Env** (`src/env.ts`): `EvalEnvLive` mirrors `main.ts`'s composition but swaps Postgres for an in-memory `ConversationStore` (`support/inMemoryConversationStore.ts`, replicating the position/checkpoint fold semantics) so evals need no Docker. `support/workspace.ts` gives `withTempWorkspace` (acquire/release temp dirs); `support/coder.ts` (`runCoder`) stands up a real coder agent over a temp repo and reports the tools it called + final text + read-back files.
- **Config injection** (`src/config/`): a `RunConfig` pins `{main, fast, judge, promptVariant, maxSteps}`; `makeEvalEnv(config?)` swaps in a `FixedSettingsStoreLive` (its `load()` ignores disk) so a `--config matrix.json` run is the independent variable A/B'd against a baseline. No config ŌćÆ today's `LocalSettingsStoreLive`.
- **Suites** (`src/suites/*.eval.ts`): `handoff`, `tool-selection`, `coder-edit` (originals) + `whole-task` (bug-fix / multi-file / refactor / failing-test / read-only Q&A through the FULL loop), and three **fast-model** suites calling the fast-tier use cases directly ŌĆö `judge-approval` (labelled allow/prompt accuracy), `compaction-digest` (`compressToolResults` fidelity), `session-title` (`generateSessionTitle` quality).
- **Run**: `bun run eval [name ŌĆ”] [--config f] [--main m] [--fast m] [--json]` (`src/run.ts`). Gated on `hasKey` ŌĆö no key ŌåÆ skip cleanly. Each run is collected **in-memory** and the report is **built from the spans** (`trace/process.ts` ŌåÆ `trace/report.ts`); set `OTEL_EXPORTER_OTLP_ENDPOINT` and it also streams to Grafana. Unit tests (`bun test`) cover the framework, the store fold contract, and the pure trace processor (synthetic spans), with no LLM/Docker.

## Observability (traces + metrics)

Trace-first, by design: **production code only annotates** OpenTelemetry spans and records a few Effect `Metric`s ŌĆö it never builds a metrics store. The waterfall is the full execution: `agent.run` (one per **user message** ŌĆö carries `agent.conversation_id`, an `agent.prompt` excerpt, `agent.model`, run-total tokens) ŌåÆ `agent.turn` ŌåÆ `llm.generate` (model + `gen_ai.usage.*` + `gen_ai.cost_usd` + `gen_ai.cache_hit_ratio` + finish reason, plus the clipped `gen_ai.prompt`/`gen_ai.completion` text ŌĆö captured whenever telemetry is on, no separate flag) ŌåÆ **`agent.tool`** (one per tool call ŌĆö `agent.tool.name` / `.ok` / `.args_summary`, the seam in `recoverMalformedToolCalls`) and **`agent.subagent`** (a spawned run's subtree ŌĆö node/depth/folder), plus the helper spans `agent.approval.judge` / `agent.compaction.digest` / `agent.title`. Metrics (`core/src/telemetry/metrics.ts`): `gen_ai_tokens_total`, `gen_ai_calls_total`, **`gen_ai_cost_usd_total`** (priced via the same `costUsd` catalogue the evals use), `agent_turns_total`, `agent_turn_latency_ms`, `agent_tool_calls_total`, **`agent_errors_total{kind=run|turn|tool|llm,error}`** (RED's E), `approval_verdicts_total`. **Bounded tags only** ŌĆö conversation/node/run id + prompt live on **spans**, never on a metric. Everything is **inert without a tracer** ŌĆö zero overhead un-exported.

- **Export** is one layer at the edge: `OtlpTelemetryLive` (`adapters/src/telemetry/otlp.ts`, Effect-native `Otlp.layerJson` over HTTP ŌĆö no heavy OTel SDK in the CLI bundle). It's wired through a `Layer.unwrapEffect` in `main.ts` that reads the **`Settings.telemetry`** flag (the SOLE switch, `:set telemetry on`; schema default off) and returns `OtlpTelemetryLive` or `Layer.empty` ŌĆö no env var enables export. Resource: `service.name=efferent` + `deployment.environment=production`; the endpoint defaults to `http://localhost:4318` (`OTEL_EXPORTER_OTLP_ENDPOINT` overrides WHERE, not WHETHER).
- **Eval / prod separation** ŌĆö one stack, logical split. Evals export under `service.name=efferent-evals` (ŌåÆ Prometheus `job`) + `deployment.environment=eval` + a `resource.eval.run_id`, into a **separate Grafana folder**; prod dashboards filter `{job="efferent"}`, eval dashboards `{job="efferent-evals"}`. Evals still **build their report FROM in-memory spans** (`trace/process.ts`) and only ship to the stack when `OTEL_EXPORTER_OTLP_ENDPOINT` is set.
- **Grafana dashboards** (`grafana/otel-lgtm`; auto-provisioned from `observability/grafana/dashboards/{production,evals}/` via two folder providers).
